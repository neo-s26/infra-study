# 第1章 クラウドの概念｜試験型ガイド

## 1. この章で問われること

この章は、AZ-900の中でも「クラウドとは何か」「オンプレミスと何が違うか」「IaaS / PaaS / SaaS の違い」を理解する章。

試験では、細かい歴史よりも次の観点が問われやすい。

- クラウドとオンプレミスの違い
- クラウドを利用するメリット
- CAPEX と OPEX の違い
- 高可用性・耐障害性・スケーラビリティ・弾力性・機敏性
- パブリック / プライベート / ハイブリッドクラウドの違い
- IaaS / PaaS / SaaS の責任範囲
- ユースケースに合うクラウドモデル・サービスモデルの選択

---

## 2. 5分で見直す要点

### クラウドとは

クラウドとは、クラウドサービスプロバイダーが保有・管理するサーバー、ストレージ、ネットワークなどを、インターネット経由で利用する仕組み。

ユーザーや企業は、自社で物理サーバーを持たなくても、必要なITリソースを利用できる。

試験では次のように押さえる。

- クラウド：外部のクラウド事業者のリソースを利用する
- オンプレミス：自社内・自社管理の環境でサーバーやネットワークを運用する
- クラウドはインターネット経由で利用できる
- 必要なときに必要なリソースを作成できる
- 物理機器の購入や設置を利用者が行わなくてよい

---

### オンプレミスとは

オンプレミスは、企業が自社のサーバールームやデータセンターにサーバー、ストレージ、ネットワーク機器などを設置して運用する形態。

クラウドと比較すると、以下の特徴がある。

- ハードウェアを自社で購入する
- 設置・配線・保守・交換を自社側で行う
- 初期費用が大きくなりやすい
- 自由度は高いが、管理負担も大きい
- 障害対策や拡張にも物理的な作業が必要になる

---

### 仮想化

クラウドでは仮想化技術が使われている。

仮想化とは、1台の物理サーバー上に複数の仮想マシンを動かす技術。

試験では次のように押さえる。

- 物理サーバーを効率よく使える
- 複数の仮想マシンを1台の物理サーバー上で動かせる
- 仮想マシンにはOSやアプリケーションをインストールできる
- 利用者は、物理サーバーか仮想マシンかを意識せずにサービスを利用できる
- サーバー台数削減やコスト削減につながる

---

### ネットワークの仮想化

クラウドでは、サーバーだけでなくネットワークも仮想化される。

オンプレミスでは、ネットワーク変更にルーター、スイッチ、LANケーブルなどの物理作業が必要になる。

クラウドでは、管理ポータルから仮想ネットワークを作成・変更できる。

試験では次のように押さえる。

- 仮想ネットワークをソフトウェアで構成できる
- 物理的な配線作業が不要
- ネットワークを複数作成できる
- ネットワークをセグメントに分割できる
- 管理ポータルから構成変更できる

---

### オンデマンドセルフサービス

オンデマンドセルフサービスとは、利用者が必要なときに自分でリソースを作成・変更できる仕組み。

クラウドでは、管理ポータルから仮想マシンや仮想ネットワークを自動的に作成できる。

試験では、

- 必要なときに
- 利用者自身が
- 管理ポータルなどから
- 人手を介さず
- ITリソースを作成できる

というイメージで覚える。

---

## 3. クラウドを利用するメリット

### コスト削減

クラウドでは、事前に物理サーバーなどのハードウェアを購入する必要がない。

そのため、初期費用を抑えやすい。

また、基本的には利用した分だけ支払う従量課金になる。

試験では、CAPEX と OPEX の違いが重要。

| 用語 | 意味 | 覚え方 |
|---|---|---|
| CAPEX | 資本的支出・設備投資 | サーバー購入などの初期投資 |
| OPEX | 運用コスト | 使った分だけ払う運用費 |

クラウドでは、CAPEXを減らし、OPEX中心にできる。

---

### 高可用性

高可用性とは、システムが止まりにくい状態のこと。

業務に重要なサーバーでは、障害が起きてもサービスを継続できる仕組みが必要になる。

試験では次の用語が出やすい。

| 用語 | 意味 |
|---|---|
| 可用性 | システムを利用できる状態が保たれていること |
| 高可用性 | 停止時間や停止頻度が少ないこと |
| フォールトトレランス | 障害が起きてもサービスを継続できる性質 |
| フェールオーバー | 障害時に待機系へ切り替えること |
| クラスター | 複数台のサーバーを1つのシステムのように構成すること |

---

### 可用性ゾーン

可用性ゾーンは、同じリージョン内にある複数の独立したデータセンターを使って障害に備える仕組み。

1つのデータセンターに障害が発生しても、別のゾーンに配置したリソースでサービスを継続できる。

試験では次のように押さえる。

- リージョン内のデータセンター障害対策
- 同一リージョン内でリソースを分散配置する
- データセンター単位の障害に備える
- リージョン全体の障害には対応できない

---

### geo分散

geo分散は、災害対策のために離れた地域、つまり別リージョンにアプリやデータを分散配置すること。

可用性ゾーンではリージョン全体の障害には対応できないため、別リージョンへの複製が必要になる。

試験では次のように押さえる。

- リージョン全体の障害に備える
- 別リージョンにデータや仮想マシンを複製する
- 大規模災害や地域障害への対策
- Azure Site Recovery は仮想マシンを別リージョンへ複製するサービス

---

### Azure Site Recovery

Azure Site Recovery、ASRは、仮想マシンを別リージョンに複製し、障害時にフェールオーバーできるサービス。

試験では、

- 災害対策
- リージョン障害への備え
- 仮想マシンの複製
- セカンダリリージョンへのフェールオーバー

というキーワードで出やすい。

---

### スケーラビリティ

スケーラビリティとは、需要の変化に合わせてITリソースを増減できる能力。

クラウドでは、管理ポータルから仮想マシンのスペック変更や台数変更ができる。

試験では、次の2種類の違いを意識する。

| 種類 | 意味 | 例 |
|---|---|---|
| スケールアップ | 1台の性能を上げる | CPUやメモリを増やす |
| スケールアウト | 台数を増やす | WebサーバーのVMを増やす |

---

### 弾力性

弾力性、Elasticityとは、負荷に応じてリソースを動的に増減できる性質。

スケーラビリティが「増減できる能力」だとすると、弾力性は「需要に応じて柔軟に増減できる性質」と考える。

Azure Virtual Machine Scale Setsを使うと、仮想マシンの台数を自動的に増減できる。

試験では次のように押さえる。

- アクセス増加時に台数を増やす
- アクセス減少時に台数を減らす
- パフォーマンスデータに基づいて自動スケールできる
- スケジュールに基づいた自動スケールもできる

---

### 機敏性

機敏性、Agilityとは、必要なシステムをすばやく構築・変更できる性質。

オンプレミスでは、サーバー購入、納品、設置、OS設定などに時間がかかる。

クラウドでは、管理ポータルから数分でリソースを作成できる。

試験では、

- すばやく構築できる
- すばやく変更できる
- ハードウェア購入が不要
- ビジネス要求にすぐ対応できる

という意味で押さえる。

---

## 4. クラウドモデル

### パブリッククラウド

パブリッククラウドは、クラウドサービスプロバイダーが提供するサービスをインターネット経由で利用するモデル。

Azure、AWS、GCPなどが該当する。

特徴は次のとおり。

- 自社でハードウェアを購入しない
- クラウド事業者のリソースを利用する
- 基本は従量課金
- CAPEXを抑えられる
- OPEXが発生する
- 管理負担を減らしやすい
- 一般的には他の組織と物理リソースを共有する

---

### 専用パブリッククラウド

専用パブリッククラウドは、クラウドサービスプロバイダーの環境を使うが、物理サーバーを他の利用者と共有しないモデル。

Azureでは、Azure Dedicated Hostが該当する。

試験では次のように押さえる。

- 1つの組織が物理サーバーを占有する
- 他の利用者とハードウェアを共有しない
- コンプライアンス要件がある場合に使う
- 共有型より高価になりやすい

---

### プライベートクラウド

プライベートクラウドは、自社のデータセンターなどに構築した仮想化環境を、自社ユーザーが利用するモデル。

特徴は次のとおり。

- 自社でハードウェアを購入・管理する
- 他の組織とリソースを共有しない
- セキュリティやカスタマイズ性を高めやすい
- 導入コストと管理負担が大きい
- 仮想化によりスケーラビリティを得られる

---

### ハイブリッドクラウド

ハイブリッドクラウドは、パブリッククラウドとプライベートクラウド、またはオンプレミス環境を組み合わせて使うモデル。

試験ではユースケースで出やすい。

例：

- 機密性の高いデータはオンプレミスに置く
- 一般的なアプリやデータはクラウドに置く
- オンプレミスに負荷がかかったらクラウドを使う
- 既存環境を残しつつクラウドを活用する

---

## 5. サービスモデル

### IaaS

IaaSは、Infrastructure as a Serviceの略。

サーバー、ストレージ、ネットワークなどのインフラをサービスとして利用するモデル。

Azureでは、Azure Virtual Machinesが代表例。

試験では次のように押さえる。

- 仮想マシンを作成できる
- OSや構成の自由度が高い
- 利用者がOSやアプリの管理を行う
- クラウド事業者は物理サーバーや仮想化基盤を管理する
- 自由度は高いが、管理負担も大きい

---

### PaaS

PaaSは、Platform as a Serviceの略。

アプリケーションを実行するためのプラットフォームを提供するモデル。

Azureでは、Azure App Service、Azure SQL Database、Azure Cosmos DB、ストレージアカウントなどが代表例。

試験では次のように押さえる。

- 利用者はアプリやデータに集中できる
- OSやミドルウェアの管理負担が少ない
- IaaSより自由度は下がる
- IaaSより管理負担は小さい
- 開発・実行環境をすばやく利用できる

---

### SaaS

SaaSは、Software as a Serviceの略。

アプリケーションそのものをサービスとして利用するモデル。

Microsoft 365、Intune、Dynamics 365などが代表例。

試験では次のように押さえる。

- 利用者は完成済みのアプリを使う
- サーバーやOS、アプリ基盤を構築しない
- ライセンス契約ですぐ使える
- 管理負担が最も小さい
- 自由度は最も低い

---

## 6. 混同しやすい比較

### クラウドとオンプレミス

| 比較 | クラウド | オンプレミス |
|---|---|---|
| サーバー所有 | クラウド事業者 | 自社 |
| 初期費用 | 小さい | 大きい |
| 課金 | 従量課金 | 設備投資中心 |
| 拡張 | 管理ポータルで変更しやすい | 機器購入や作業が必要 |
| 管理負担 | 小さくしやすい | 大きい |
| 構築速度 | 速い | 時間がかかる |

---

### CAPEXとOPEX

| 用語 | 内容 | 試験での見分け方 |
|---|---|---|
| CAPEX | 設備投資 | サーバー購入、機器購入、初期投資 |
| OPEX | 運用費 | 使った分だけ支払う、従量課金 |

クラウドは、CAPEXを削減してOPEXに移行しやすい。

---

### 可用性ゾーンとgeo分散

| 比較 | 可用性ゾーン | geo分散 |
|---|---|---|
| 対象 | データセンター障害 | リージョン全体の障害 |
| 範囲 | 同一リージョン内 | 離れたリージョン間 |
| 目的 | リージョン内の高可用性 | 災害対策 |
| キーワード | ゾーン、データセンター分散 | 別リージョン、災害復旧 |

---

### スケールアップとスケールアウト

| 比較 | 意味 | 例 |
|---|---|---|
| スケールアップ | 1台の性能を上げる | VMのCPUやメモリを増やす |
| スケールアウト | 台数を増やす | VMを複数台に増やす |

---

### スケーラビリティと弾力性

| 比較 | 意味 |
|---|---|
| スケーラビリティ | リソースを増減できる能力 |
| 弾力性 | 需要に応じて動的に増減できる性質 |

試験では、負荷に応じて自動で増減する話なら「弾力性」「自動スケール」が出やすい。

---

### パブリック / プライベート / ハイブリッドクラウド

| モデル | 特徴 | 向いているケース |
|---|---|---|
| パブリッククラウド | クラウド事業者の環境を利用 | コスト削減、迅速な構築 |
| プライベートクラウド | 自社専用のクラウド環境 | 高いカスタマイズ性、専用環境 |
| ハイブリッドクラウド | パブリックとプライベートを併用 | 機密データは自社、その他はクラウド |

---

### IaaS / PaaS / SaaS

| モデル | 利用者が主に管理するもの | 代表例 | 覚え方 |
|---|---|---|---|
| IaaS | OS、アプリ、データ | Azure Virtual Machines | 仮想マシンを借りる |
| PaaS | アプリ、データ | Azure App Service、Azure SQL Database | 実行基盤を借りる |
| SaaS | データや利用設定中心 | Microsoft 365、Intune、Dynamics 365 | 完成アプリを使う |

管理負担は、IaaSが大きく、SaaSが小さい。  
自由度は、IaaSが高く、SaaSが低い。

---

## 7. 試験での聞かれ方

### クラウドのメリットを選ばせる問題

「サーバー購入を避けたい」「初期費用を抑えたい」  
→ CAPEX削減、従量課金、OPEX

「急にWebサーバーが必要になった」  
→ 機敏性

「アクセス増加に合わせて台数を増やしたい」  
→ スケーラビリティ、自動スケール、弾力性

「障害時にもサービスを継続したい」  
→ 高可用性、フォールトトレランス、フェールオーバー

---

### 障害対策を選ばせる問題

「1つのデータセンター障害に備えたい」  
→ 可用性ゾーン

「リージョン全体の障害や大災害に備えたい」  
→ geo分散、別リージョン、Azure Site Recovery

「仮想マシンを別リージョンに複製したい」  
→ Azure Site Recovery

---

### クラウドモデルを選ばせる問題

「自社でサーバーを持ちたくない」  
→ パブリッククラウド

「他社と物理リソースを共有したくない」  
→ 専用パブリッククラウド、Azure Dedicated Host

「自社専用環境で高いカスタマイズ性が必要」  
→ プライベートクラウド

「機密データは自社、その他はクラウド」  
→ ハイブリッドクラウド

---

### サービスモデルを選ばせる問題

「仮想マシンを自由に構成したい」  
→ IaaS

「OSやミドルウェア管理を避け、アプリ開発に集中したい」  
→ PaaS

「完成済みのアプリをすぐ使いたい」  
→ SaaS

「メールサーバーを構築せずメール機能を使いたい」  
→ SaaS、Microsoft 365 / Exchange Online

「SQL Serverを自分で構築せずデータベースを使いたい」  
→ PaaS、Azure SQL Database

---

## 8. 最後に覚えるキーワード

- クラウド：クラウド事業者のITリソースをインターネット経由で利用
- オンプレミス：自社でサーバーやネットワークを設置・管理
- 仮想化：1台の物理サーバー上で複数の仮想マシンを動かす
- オンデマンドセルフサービス：必要なときに自分でリソースを作成できる
- CAPEX：設備投資
- OPEX：運用費、従量課金
- 高可用性：止まりにくいこと
- フォールトトレランス：障害が起きても継続できる性質
- フェールオーバー：障害時の切り替え
- 可用性ゾーン：同一リージョン内のデータセンター障害対策
- geo分散：別リージョンを使う災害対策
- Azure Site Recovery：仮想マシンを別リージョンに複製して復旧
- スケーラビリティ：リソースを増減できる能力
- 弾力性：需要に応じて動的にリソースを増減できる性質
- 機敏性：すばやく構築・変更できる性質
- パブリッククラウド：クラウド事業者の環境を利用
- プライベートクラウド：自社専用のクラウド環境
- ハイブリッドクラウド：オンプレミスやプライベートとパブリックを併用
- IaaS：仮想マシンなどのインフラを利用
- PaaS：アプリ実行基盤を利用
- SaaS：完成済みアプリを利用
