# 第4章 Azureのコンピューティングサービス｜試験型ガイド

## 1. この章で問われること

この章は、Azureでアプリケーションや処理を実行するためのコンピューティングサービスを理解する章。

試験では、細かい構築手順よりも次の観点が問われやすい。

- Azure Virtual Machinesの特徴
- 仮想マシンのサイズと関連リソース
- 可用性セットと可用性ゾーンの違い
- Azure Virtual Machine Scale Setsの役割
- Azure App Serviceの特徴
- コンテナーサービスの使い分け
- Azure Functionsのサーバーレスとイベント駆動
- Azure Virtual Desktopの用途

---

## 2. 5分で見直す要点

### Azure Virtual Machines

Azure Virtual Machinesは、Azure上に仮想マシンを作成するIaaSのコンピューティングサービス。

ユーザーは仮想マシンのOS、サイズ、構成、メンテナンスを管理する。

試験では次のように押さえる。

- IaaSの代表的なサービス
- OSやスペックを自由に選べる
- 物理ハードウェアはMicrosoftが管理する
- ゲストOSやアプリケーションは利用者が管理する
- 自由度は高いが、管理負荷も高い

---

### ホストOSとゲストOS

| 用語 | 意味 |
|---|---|
| ホストOS | 物理サーバー側のOS |
| ゲストOS | 仮想マシン上で動くOS |

Azure VMでは、利用者が主に意識するのはゲストOS。

---

### 仮想マシンのサイズ

仮想マシンのサイズとは、VMに割り当てるCPU、メモリ、ディスク数などのスペック。

サイズによって性能と料金が変わる。

| 種類 | 向いている用途 |
|---|---|
| 汎用 | CPUとメモリのバランスが必要な一般用途 |
| コンピューティング最適化 | 高いCPU性能が必要な処理 |
| メモリ最適化 | 大量メモリが必要な処理 |
| ストレージ最適化 | 高いディスク性能が必要な処理 |
| GPU | 画像処理、動画編集、機械学習 |
| HPC | 非常に高い計算能力が必要な処理 |

試験では、用途に応じてVMサイズを選ぶと押さえる。

---

### 仮想マシンの関連リソース

仮想マシンは、単体ではなく複数の関連リソースと組み合わせて作成される。

主な関連リソース：

- 仮想ネットワーク
- ネットワークインターフェイス
- ディスク
- ネットワークセキュリティグループ
- パブリックIPアドレス

必須・任意の考え方：

| リソース | 必須 / 任意 | 役割 |
|---|---|---|
| 仮想ネットワーク | 必須 | VMを接続するネットワーク |
| ネットワークインターフェイス | 必須 | VMをネットワークに接続する |
| ディスク | 必須 | OSやデータを保存する |
| ネットワークセキュリティグループ | 任意 | 通信を制御する |
| パブリックIPアドレス | 任意 | インターネットから直接接続する |

NSGは任意だが、セキュリティ上は構成することが重要。

---

### 可用性セット

可用性セットは、同じリージョン内で複数の仮想マシンを分散配置し、停止リスクを減らす仕組み。

可用性セットは次の2つで構成される。

| 種類 | 目的 | 内容 |
|---|---|---|
| 障害ドメイン | ハードウェア障害対策 | ラック単位の障害に備えてVMを分散 |
| 更新ドメイン | メンテナンス対策 | メンテナンス時にVMが同時停止しないように分散 |

試験では、可用性セットは同一リージョン内のVM停止リスクを分散する仕組みと押さえる。

---

### 障害ドメイン

障害ドメインは、電源やネットワーク機器などを共有するラック単位の障害に備えるグループ。

試験では次のように押さえる。

- ハードウェア障害対策
- ラック単位の障害に備える
- VMを複数の障害ドメインに分散する
- 1つの障害ドメインに障害が起きても、他のVMは稼働できる

---

### 更新ドメイン

更新ドメインは、Azure基盤のメンテナンスに備えるグループ。

試験では次のように押さえる。

- メンテナンス対策
- VMの同時停止を防ぐ
- メンテナンスは更新ドメイン単位で行われる
- 複数VMを別々の更新ドメインに分散する

---

### 可用性ゾーン

可用性ゾーンは、1つのリージョン内の異なるデータセンターにVMを分散配置する仕組み。

試験では次のように押さえる。

- データセンター障害対策
- 同一リージョン内の複数ゾーンにVMを分散する
- 1つのゾーンで障害が発生しても他のゾーンのVMは稼働できる
- 可用性セットよりもデータセンター障害に強い

---

## 3. Azure Virtual Machine Scale Sets

### Azure Virtual Machine Scale Setsとは

Azure Virtual Machine Scale Setsは、同じ構成の複数の仮想マシンをまとめて作成・管理するサービス。

試験では次のように押さえる。

- 複数VMをグループ化して管理する
- 同じ構成のVMをまとめて作成できる
- スケールセット全体に設定を適用できる
- 負荷に応じてVM台数を自動増減できる
- 冗長化や負荷分散に向く

---

### 垂直スケーリング

垂直スケーリングは、1台の仮想マシンの性能を増減させること。

| 操作 | 意味 |
|---|---|
| スケールアップ | CPUやメモリを増やして性能を上げる |
| スケールダウン | CPUやメモリを減らして性能を下げる |

VMサイズを変更するイメージ。

---

### 水平スケーリング

水平スケーリングは、仮想マシンの台数を増減させること。

| 操作 | 意味 |
|---|---|
| スケールアウト | VMの台数を増やす |
| スケールイン | VMの台数を減らす |

Virtual Machine Scale Setsの自動スケールは、水平スケーリングの代表例。

---

## 4. Azure App Service

### Azure App Serviceとは

Azure App Serviceは、Webアプリケーションを実行するためのPaaSのコンピューティングサービス。

Webサーバーやランタイムなどの基盤はMicrosoftが管理するため、利用者はアプリケーション開発と配置に集中できる。

試験では次のように押さえる。

- PaaSのWebアプリ実行サービス
- Webアプリケーションの配置・運用に使う
- OSやWebサーバーの管理負荷を減らせる
- C#、Javaなどで作成したWebアプリを配置できる
- インフラ管理よりアプリ開発に集中したい場合に向く

---

### App Serviceプラン

App Serviceを使うには、App Serviceプランが必要。

App Serviceプランは、Webアプリを実行する土台となる環境の構成。

設定する主な内容：

- OSの種類
- リージョン
- 仮想マシンの数
- 仮想マシンのサイズ
- 価格レベル

価格レベルの例：

| 価格レベル | 特徴 |
|---|---|
| Free | 無料だが本番向け機能は少ない |
| Basic | 基本的な運用向け |
| Standard | 自動スケールなどが利用可能 |
| Premium | 高機能。可用性ゾーン利用などに対応 |
| Isolated | 分離された高機能環境 |

試験では、価格レベルによって使える機能と料金が変わると押さえる。

---

### App Serviceの自動スケール

App Serviceにも、負荷に応じてインスタンス数を増減する自動スケール機能がある。

試験では次のように押さえる。

- Webアプリの負荷に応じて台数を増減できる
- 負荷が高いときは台数を増やす
- 負荷が低いときは台数を減らす
- コスト最適化に役立つ
- Standard以上の価格レベルで利用可能

---

## 5. コンテナーサービス

### コンテナーとは

コンテナーは、アプリケーションと実行環境をまとめて動かす仮想化技術。

コンテナーイメージをもとにコンテナーを作成し、さまざまな環境で実行できる。

試験では次のように押さえる。

- アプリと実行環境をまとめる
- コンテナーイメージから作成する
- レジストリにイメージを保存する
- 移植性が高い
- 仮想マシンより軽量で起動が速い

---

### コンテナーと仮想マシンの違い

| 比較 | コンテナー | 仮想マシン |
|---|---|---|
| OS | ホストOSのカーネルを共有 | ゲストOSを持つ |
| 重さ | 軽量 | 重め |
| 起動 | 速い | 比較的遅い |
| 単位 | アプリと実行環境 | OSを含む仮想コンピューター |

コンテナーはOS全体を持たないため、VMより軽量に動作する。

---

### レジストリ

レジストリは、コンテナーイメージを保存する場所。

代表例：

- Docker Hub
- Azure Container Registry

試験では、レジストリはコンテナーイメージの格納場所と覚える。

---

### Web App for Containers

Web App for Containersは、App Service上でコンテナーを実行するサービス。

試験では次のように押さえる。

- App Serviceでコンテナーを動かす
- Webアプリのコンテナー運用に向く
- App Serviceの自動スケールなどを活用できる
- サーバーのメンテナンス負荷を減らせる
- App Serviceプランの料金が発生する

---

### Azure Container Instances

Azure Container Instances、ACIは、単一コンテナーを素早く簡単に実行できるサービス。

試験では次のように押さえる。

- 最も簡単にコンテナーを実行できる
- サーバー管理が不要
- 秒単位で課金される
- 単一コンテナーの実行に向く
- 短時間処理やバッチ処理に向く
- 複数コンテナーの高度な管理や自動スケールには向かない

---

### Azure Kubernetes Service

Azure Kubernetes Service、AKSは、複数のコンテナーを管理・自動化するサービス。

試験では次のように押さえる。

- 複数コンテナーの運用管理に向く
- コンテナーの配置を管理できる
- 自動スケールと負荷分散ができる
- 正常性監視やメンテナンスを支援する
- コンテナーオーケストレーションのサービス

---

### コンテナーオーケストレーション

コンテナーオーケストレーションとは、複数のコンテナーの配置、スケール、負荷分散、監視などを自動管理すること。

AKSは、Azure上でKubernetesを利用するためのマネージドサービス。

---

## 6. Azure Functions

### Azure Functionsとは

Azure Functionsは、サーバーレスコンピューティングに分類されるサービス。

サーバーレスとは、サーバーが存在しないという意味ではなく、利用者がサーバーを意識しなくてよいという意味。

試験では次のように押さえる。

- サーバーレスの代表的なサービス
- 利用者はサーバーのスペックを管理しない
- コードの実装に集中できる
- 処理実行時間や実行回数に応じて課金されるプランがある
- 短い処理やイベント処理に向く

---

### イベント駆動とトリガー

Azure Functionsはイベント駆動型のサービス。

何かのイベントが発生したことをきっかけに処理を開始する。

イベントの例：

- ストレージに画像ファイルが格納された
- FunctionsのURLが呼び出された
- Cosmos DBにデータが格納された

トリガーとは、Functionsを起動するきっかけの設定。

試験では、イベント発生 → トリガー → 関数実行、の流れで押さえる。

---

### Azure Functionsで使える言語

Azure Functionsでは、複数の言語で関数を作成できる。

代表例：

- C#
- JavaScript
- Java
- PowerShell
- Python
- TypeScript

試験では、好みの言語でイベント処理のコードを書けるサービスと押さえる。

---

## 7. Azure Virtual Desktop

### Azure Virtual Desktopとは

Azure Virtual Desktop、AVDは、Azure上で仮想デスクトップや仮想アプリケーションを提供するサービス。

ユーザーはローカルPCからAzure上のデスクトップ環境に接続して利用する。

試験では次のように押さえる。

- デスクトップ仮想化サービス
- Azure上のVMでデスクトップ環境を提供する
- ローカルPCにデータを置かずに利用できる
- アプリケーションだけを仮想化して配信することもできる
- 端末管理やセキュリティ向上に役立つ

---

### AVDのメリット

AVDの主なメリット：

- ハイスペックなPCがなくても重いアプリを利用できる
- データをAzure上に置けるため、端末故障時の影響を抑えられる
- ローカルPC紛失時の情報漏えいリスクを減らせる
- OSやアプリケーションをAzure上で一元管理できる
- 管理者の端末管理負荷を減らせる

---

### マルチセッション

マルチセッションは、1台の仮想マシンに複数ユーザーが同時接続できる機能。

試験では次のように押さえる。

- Windows 10 / 11 Enterpriseマルチセッションで利用できる
- 1台のVMを複数ユーザーで共有できる
- VM台数を減らせる
- コスト削減につながる

---

### ホストプール

ホストプールとは、AVD環境で同じ構成の仮想マシンをまとめたグループ。

ホストプールでは、複数ユーザーの接続をVMへ分散できる。

分散方法：

| 方法 | 内容 |
|---|---|
| 均等分散 | 複数VMにユーザー接続を均等に分散する |
| セッション上限分散 | 1台のVMが上限に達したら次のVMへ接続する |

試験では、ホストプールはAVDのVM群をまとめて管理する単位と押さえる。

---

## 8. 混同しやすい比較

### 可用性セットと可用性ゾーン

| 比較 | 可用性セット | 可用性ゾーン |
|---|---|---|
| 範囲 | 同一リージョン内 | 同一リージョン内 |
| 主な対策 | ラック障害・メンテナンス | データセンター障害 |
| 分散単位 | 障害ドメイン、更新ドメイン | ゾーン |
| キーワード | 同時停止を防ぐ | データセンター分散 |

---

### 垂直スケーリングと水平スケーリング

| 比較 | 意味 | 操作 |
|---|---|---|
| 垂直スケーリング | 1台の性能を増減 | スケールアップ / スケールダウン |
| 水平スケーリング | 台数を増減 | スケールアウト / スケールイン |

---

### Azure VMとApp Service

| 比較 | Azure VM | App Service |
|---|---|---|
| サービスモデル | IaaS | PaaS |
| 管理対象 | OSやアプリも利用者が管理 | アプリ中心に管理 |
| 自由度 | 高い | VMより低い |
| 管理負荷 | 高い | 低い |
| 向いている用途 | OSレベルで自由に構成したい | Webアプリを楽に運用したい |

---

### ACIとAKS

| 比較 | ACI | AKS |
|---|---|---|
| 用途 | 単一コンテナーを簡単に実行 | 複数コンテナーを本格管理 |
| 管理 | シンプル | 高度 |
| スケール | 限定的 | 自動スケールや負荷分散 |
| 向いている処理 | 短時間処理、簡単なバッチ | 大規模なコンテナー運用 |

---

### App ServiceとAzure Functions

| 比較 | App Service | Azure Functions |
|---|---|---|
| 用途 | Webアプリを常時運用 | イベント発生時に処理を実行 |
| 実行形態 | アプリを配置して実行 | 関数単位でコードを実行 |
| 管理 | PaaS | サーバーレス |
| 向いている処理 | Webサイト、Web API | 画像処理、通知、短時間処理 |

---

## 9. 試験での聞かれ方

「OSや構成を自由に選んでサーバーを作りたい」  
→ Azure Virtual Machines

「同じ構成のVMを複数台まとめて管理し、自動で台数を増やしたい」  
→ Azure Virtual Machine Scale Sets

「Webアプリを運用したいが、OSやWebサーバー管理を減らしたい」  
→ Azure App Service

「WebアプリのコンテナーをApp Serviceで動かしたい」  
→ Web App for Containers

「単一コンテナーを短時間だけ簡単に実行したい」  
→ Azure Container Instances

「複数コンテナーをスケールや負荷分散込みで管理したい」  
→ Azure Kubernetes Service

「イベント発生時だけコードを実行したい」  
→ Azure Functions

「利用者にAzure上のデスクトップ環境を提供したい」  
→ Azure Virtual Desktop

「1台のVMに複数ユーザーを接続させたい」  
→ AVDのマルチセッション

「VMをラック障害やメンテナンスから守りたい」  
→ 可用性セット

「VMをデータセンター障害から守りたい」  
→ 可用性ゾーン

---
