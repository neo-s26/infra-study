# 第6章 Azureのストレージサービス｜試験型ガイド

## 1. この章で問われること

この章は、Azureでデータを保存・管理・移行するためのストレージサービスを理解する章。

試験では、細かい操作手順よりも次の観点が問われやすい。

- ストレージアカウントの役割
- BLOB / Files / Queue / Table の違い
- Standard / Premium の違い
- ストレージアカウントの冗長化オプション
- BLOBコンテナーとBLOBアクセス層
- Azure FilesとAzure File Syncの違い
- サービスエンドポイントの役割
- Storage Explorer / AzCopy / Azure File Syncの使い分け
- Azure MigrateとAzure Data Boxの使い分け

---

## 2. 5分で見直す要点

### ストレージアカウント

ストレージアカウントは、Azureでデータを保存するためのPaaSのストレージサービス。

HTTPまたはHTTPS経由でアクセスでき、非常に大量のデータを格納できる。

試験では次のように押さえる。

- Azureの代表的なストレージサービス
- PaaSなのでハードウェア管理は不要
- HTTP / HTTPS経由でアクセスできる
- 既定の最大容量は5PiB
- 保存量、アクセス回数、冗長性、リージョンなどで料金が変わる

---

### ストレージアカウントの料金に関わる要素

ストレージアカウントの料金は、保存容量だけで決まるわけではない。

主な要素：

| 要素 | 内容 |
|---|---|
| リージョン | 作成するリージョンによって料金が変わる |
| パフォーマンス | Standard / Premiumで料金と性能が変わる |
| データ量 | 保存するGB単位の容量 |
| アクセス回数 | 読み取り・書き込みなどの操作回数 |
| 冗長性 | LRS、ZRS、GRSなどの冗長化方式 |
| データ転送 | リージョン間コピーなどで発生 |

---

### Azure BLOB

Azure BLOBは、大きなバイナリデータを格納するサービス。

画像、動画、PDF、Excel、圧縮ファイルなど、ファイル形式の大きなデータに向いている。

試験では次のように押さえる。

- BLOB = Binary Large Object
- 大容量ファイルの保存に向く
- 画像、動画、PDF、バックアップデータなどを格納できる
- Webアプリから画像やファイルを読み取る用途に使える
- VMディスクは現在はマネージドディスク利用が推奨

---

### Azure Files

Azure Filesは、Azure上に共有フォルダーを作成できるファイル共有サービス。

オンプレミスのファイルサーバーのように利用できる。

試験では次のように押さえる。

- ファイル共有サービス
- 共有フォルダーをAzure上に作成できる
- ネットワークドライブとして割り当てできる
- Windows、Linux、macOSから利用できる
- PaaSなのでファイルサーバーのハードウェア管理が不要

---

### Azure Queue

Azure Queueは、アプリケーション間でメッセージをやり取りするためのサービス。

送信側と受信側の間にキューを挟むことで、負荷を分散しながら処理できる。

試験では次のように押さえる。

- メッセージングサービス
- メッセージを一時的に保存する
- アプリケーション間の連携に使う
- 受信側の処理負荷を平準化できる
- 非同期処理に向く

---

### Azure Table

Azure Tableは、表形式でデータを格納するサービス。

キーと値を紐づけて保存し、キーを指定してデータを取り出す。

試験では次のように押さえる。

- テーブル形式でデータを保存する
- キーと値で管理する
- 一般的なリレーショナルDBよりシンプル
- 分散保存や高速アクセスに向く
- 複雑なSQL処理よりシンプルなデータ格納向け

---

### Azure SQL Database

Azure SQL Databaseは、SQL ServerをAzure上で利用できるPaaSのデータベースサービス。

試験では次のように押さえる。

- リレーショナルデータベース
- 行と列で構成された表形式のデータを扱う
- SQLでデータを検索・操作する
- SQL Serverを自分で構築しなくてよい
- バックアップやメンテナンスの管理負荷が低い

---

### Azure Cosmos DB

Azure Cosmos DBは、世界中のリージョンにデータを分散配置できるPaaSのデータベースサービス。

試験では次のように押さえる。

- グローバル分散に強いデータベース
- JSON形式やキーと値形式などの半構造化データを扱える
- ユーザーに近いリージョンへデータを配置できる
- 応答速度を高めたいアプリに向く
- PaaSなので管理負荷が低い

---

## 3. ストレージアカウントの種類

### StandardとPremium

ストレージアカウントのパフォーマンスには、StandardとPremiumがある。

| 種類 | 内部的なストレージ | 特徴 |
|---|---|---|
| Standard | HDD | 汎用的でコストを抑えやすい |
| Premium | SSD | 高速だがコストが高い |

試験では、性能重視ならPremium、汎用・低コストならStandardと押さえる。

---

### ストレージアカウントの種類

| 種類 | 特徴 |
|---|---|
| Standard 汎用 v2 | BLOB、Files、Queue、Tableを利用できる。ほとんどの用途で推奨 |
| Premium ブロック BLOB | BLOB専用。大量BLOBデータのアップロードに最適化 |
| Premium ファイル共有 | ファイル共有専用。NFSプロトコルなどをサポート |
| Premium ページ BLOB | VMディスク向けだが、現在はマネージドディスク推奨 |

Standard 汎用 v2は、最も汎用的に使えるストレージアカウント。

---

## 4. 冗長化オプション

### ストレージの冗長化

ストレージアカウントは、データのコピーであるレプリカを保持して障害に備える。

最低でも3つのレプリカが自動的に保持される。

試験では、どこにレプリカを保持するかが重要。

---

### LRS

LRS、ローカル冗長ストレージは、1つのデータセンター内に3つのコピーを保持する。

試験では次のように押さえる。

- 1つのリージョン内の1つのデータセンターに保存
- 最も低コスト
- 冗長性は最も低い
- データセンター障害には弱い

---

### ZRS

ZRS、ゾーン冗長ストレージは、1つのリージョン内の複数ゾーンにデータを分散する。

試験では次のように押さえる。

- 1つのリージョン内の3つのゾーンに分散
- データセンター障害に備えられる
- すべてのリージョンで利用できるわけではない
- リージョン全体の障害には弱い

---

### GRS

GRS、ジオ冗長ストレージは、プライマリリージョンとセカンダリリージョンにデータを保持する。

試験では次のように押さえる。

- リージョン障害に備える
- セカンダリリージョンにもコピーを保持する
- セカンダリはリージョンペアに作成される
- 合計6つのレプリカを保持する
- 通常、セカンダリへ直接読み取りアクセスはできない

---

### GZRS

GZRS、ジオゾーン冗長ストレージは、ZRSとGRSを組み合わせた冗長化。

試験では次のように押さえる。

- プライマリはZRS構成
- セカンダリはLRS構成
- データセンター障害とリージョン障害の両方に強い
- 合計6つのレプリカを保持する

---

### RA-GRS / RA-GZRS

RAはRead Access、読み取りアクセスの意味。

RA-GRSやRA-GZRSでは、セカンダリリージョンのデータを読み取り専用で参照できる。

試験では次のように押さえる。

- セカンダリリージョンを読み取り可能にする
- 読み取り用のURLが割り当てられる
- プライマリ障害時だけでなく、平常時にもセカンダリから読める
- 書き込みは基本的にプライマリ側で行う

---

### 冗長性と料金

冗長性が高くなるほど、料金も高くなる。

基本的な順番：

- LRS
- ZRS
- GRS
- GZRS
- RA-GRS
- RA-GZRS

Standard 汎用 v2はすべての冗長化オプションをサポートする。  
Premium系は主にLRSとZRSのみをサポートする。

---

## 5. BLOBコンテナーとアクセス層

### BLOBコンテナー

BLOBコンテナーは、Azure BLOBのデータを格納するための領域。

第4章のコンテナーサービスの「コンテナー」とは別物。

試験では次のように押さえる。

- Azure BLOBを使うにはBLOBコンテナーを作成する
- BLOBコンテナー内にBLOBオブジェクトを格納する
- BLOBオブジェクトはコンテナー内のファイルのようなもの

---

### BLOBのアクセス層

BLOBのアクセス層は、データのアクセス頻度に応じてコストを最適化する仕組み。

Standard 汎用 v2のAzure BLOBで利用できる。

| アクセス層 | 向いているデータ | ストレージコスト | アクセスコスト |
|---|---|---|---|
| ホット | 頻繁にアクセスするデータ | 高い | 低い |
| クール | あまりアクセスしない短期保存データ | 中 | 中 |
| アーカイブ | ほとんどアクセスしない長期保存データ | 低い | 高い |

---

### ホットアクセス層

ホットアクセス層は、頻繁に読み書きするデータに向く。

試験では次のように押さえる。

- すぐにアクセスできる
- アクセスコストが低い
- ストレージコストが高い
- アプリケーションで頻繁に使うデータ向け

---

### クールアクセス層

クールアクセス層は、アクセス頻度が低い短期保存データに向く。

試験では次のように押さえる。

- すぐにアクセスできる
- ホットよりストレージコストが安い
- ホットよりアクセスコストが高い
- 最低30日程度保存するデータに向く
- 早期削除料金に注意

---

### アーカイブアクセス層

アーカイブアクセス層は、ほとんどアクセスしない長期保存データに向く。

試験では次のように押さえる。

- ストレージコストが最も安い
- アクセスコストが最も高い
- オフライン層に保存される
- 読み取りにはリハイドレートが必要
- 取得まで数時間かかる
- 最低180日程度保存するデータに向く
- 早期削除料金に注意

---

### リハイドレート

リハイドレートとは、アーカイブ層のBLOBをホット層またはクール層へ戻し、読み取り可能にすること。

アーカイブ層にあるBLOBは、そのままでは読み書きできない。

---

### ライフサイクル管理ポリシー

ライフサイクル管理ポリシーは、BLOBのアクセス層変更や削除を自動化する機能。

例：

- 30日更新されていないBLOBをクール層へ移動
- 180日更新されていないBLOBをアーカイブ層へ移動
- 一定期間後にBLOBを削除

試験では、アクセス層の自動変更によるコスト最適化と押さえる。

---

## 6. Azure Filesとサービスエンドポイント

### Azure Filesの利用

Azure Filesでは、Azure上にファイル共有を作成できる。

作成したファイル共有は、ネットワークドライブとして割り当てできる。

試験では次のように押さえる。

- Azure上の共有フォルダー
- オンプレミスのファイルサーバーのように使える
- Windows、Linux、macOSで利用できる
- PowerShellスクリプトなどで接続できる
- PaaSなのでファイルサーバー管理を減らせる

---

### サービスエンドポイント

サービスエンドポイントは、特定のVNetやサブネットからのみストレージアカウントへアクセスさせる仕組み。

試験では次のように押さえる。

- ストレージアカウントへのアクセス元を制限できる
- 特定のサブネットからのみアクセス許可できる
- インターネットからのアクセスを制限できる
- 通信はAzureのバックボーンネットワークを経由する
- セキュリティ向上に使う

---

## 7. ストレージ管理ツール

### Azure Storage Explorer

Azure Storage Explorerは、ストレージアカウント内のデータをGUIで管理するアプリケーション。

試験では次のように押さえる。

- GUIでストレージを操作できる
- Windows、Linux、macOSにインストールできる
- データのアップロード、ダウンロード、削除ができる
- BLOBコンテナーやファイル共有を直感的に操作できる
- Microsoft Entra ID、アクセスキー、SAS URL、接続文字列などで認証できる

---

### AzCopy

AzCopyは、ストレージアカウントへデータをコピーするためのコマンドラインツール。

試験では次のように押さえる。

- コマンドでデータコピーする
- 大量データや複数ファイルのコピーに向く
- ストレージアカウント間のコピーができる
- オンプレミスからAzureへのコピーができる
- スケジュール実行による同期にも使える

---

### Azure File Sync

Azure File Syncは、Azure FilesとオンプレミスのWindowsファイルサーバーを同期するサービス。

試験では次のように押さえる。

- Azure Filesとオンプレミスファイルサーバーを同期する
- Azure Filesでファイル共有を一元管理する
- オンプレミス側にキャッシュを置ける
- ユーザーは近い拠点のキャッシュへ高速アクセスできる
- 複数拠点で同じファイル共有を利用できる
- オンプレミスサーバー障害時もAzure側にデータが残る

---

### Azure File Syncの構成要素

| 要素 | 内容 |
|---|---|
| ストレージ同期サービス | Azure側で同期を管理するリソース |
| Azure File Syncエージェント | オンプレミスWindowsサーバーに入れるエージェント |
| 同期グループ | Azure Filesとサーバーパスを同期する単位 |
| クラウドエンドポイント | Azure Files側 |
| サーバーエンドポイント | オンプレミスWindowsサーバー側 |

---

## 8. オンプレミスからの移行

### Azure Migrate

Azure Migrateは、オンプレミス環境をAzureへ移行するための評価・移行支援プラットフォーム。

移行対象の例：

- 仮想マシン
- データベース
- Webアプリケーション
- 仮想デスクトップ

試験では次のように押さえる。

- Azure移行の評価と実行を支援する
- オンプレミス環境を検出する
- Azureへの移行適性を評価する
- 適切なVMサイズやコストを算出する
- テスト移行後に本番移行できる

---

### Azure Data Box

Azure Data Boxは、オンプレミスの大量データを物理デバイスでAzureへ移行するサービス。

試験では次のように押さえる。

- 大量データをオフラインで移行する
- 物理デバイスにデータをコピーして配送する
- ネットワークに大きな負荷をかけずに移行できる
- TB単位のデータ移行に向く
- Azureからオンプレミスへのエクスポートにも利用できる

---

### Azure Data Box Heavy

Azure Data Box Heavyは、Data Boxよりさらに大容量のデータ移行に使うサービス。

1PB規模のストレージ容量を持つデバイスを使用できる。

---

## 9. 混同しやすい比較

### BLOB / Files / Queue / Table

| サービス | 用途 |
|---|---|
| Azure BLOB | 画像、動画、PDF、バックアップなどの大容量ファイル保存 |
| Azure Files | 共有フォルダー、ファイル共有 |
| Azure Queue | アプリ間のメッセージ一時保存 |
| Azure Table | キーと値を使った表形式データ保存 |

---

### StandardとPremium

| 比較 | Standard | Premium |
|---|---|---|
| 内部ストレージ | HDD | SSD |
| 性能 | 標準 | 高速 |
| コスト | 低め | 高め |
| 用途 | 汎用 | 高性能が必要な用途 |

---

### LRS / ZRS / GRS / RA-GRS

| 冗長化 | 対策範囲 | 特徴 |
|---|---|---|
| LRS | 1データセンター内 | 最低コスト、冗長性は低い |
| ZRS | 同一リージョン内の複数ゾーン | データセンター障害に備える |
| GRS | 別リージョン | リージョン障害に備える |
| RA-GRS | 別リージョン | セカンダリを読み取り可能 |

---

### ホット / クール / アーカイブ

| 層 | 向いているデータ | 注意点 |
|---|---|---|
| ホット | 頻繁に使うデータ | ストレージコストが高い |
| クール | 低頻度アクセスの短期保存 | 早期削除料金に注意 |
| アーカイブ | 長期保管・監査データ | 読み取り前にリハイドレートが必要 |

---

### Storage Explorer / AzCopy / Azure File Sync

| ツール | 用途 |
|---|---|
| Azure Storage Explorer | GUIでストレージ内データを操作 |
| AzCopy | コマンドで大量データをコピー |
| Azure File Sync | Azure Filesとオンプレミスファイルサーバーを同期 |

---

### Azure MigrateとAzure Data Box

| 比較 | Azure Migrate | Azure Data Box |
|---|---|---|
| 目的 | サーバーやアプリの移行評価・移行 | 大量データのオフライン移行 |
| 対象 | VM、DB、Webアプリなど | ファイルや大量データ |
| 方法 | 検出、評価、レプリケーション、移行 | 物理デバイスにコピーして配送 |
| 向いているケース | オンプレミス環境をAzureへ移したい | ネットワークで送るには重すぎるデータ |

---
