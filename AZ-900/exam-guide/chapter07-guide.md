# 第7章 Azure Active Directory（Microsoft Entra ID）とセキュリティ｜試験型ガイド

## 1. この章で問われること

この章は、AzureのID管理・アクセス管理・セキュリティ対策を理解する章。

試験では、細かい設定手順よりも次の観点が問われやすい。

- 認証と認可の違い
- Azure AD（Microsoft Entra ID）の役割
- Azure ADとAD DSの違い
- Azure AD Connectの役割
- Azure AD DSの用途
- MFA、条件付きアクセス、Identity Protectionの違い
- RBACとロール割り当て
- ゼロトラストと多層防御
- Microsoft Defender for Cloudの役割

---

## 2. 5分で見直す要点

### Azure ADとMicrosoft Entra ID

Azure Active Directory、Azure ADは、現在はMicrosoft Entra IDに名称変更されている。

ただし、試験や教材ではAzure ADの名称で出る可能性がある。

試験では次のように読み替える。

- Azure AD = Microsoft Entra ID
- Azure AD MFA = Microsoft Entra MFA
- Azure AD条件付きアクセス = Microsoft Entra条件付きアクセス
- Azure AD Identity Protection = Microsoft Entra Identity Protection

機能や価格、条件、SLAは名称変更によって変わらない。

---

### 認証

認証とは、アクセスしてきた利用者が「誰なのか」を確認すること。

代表例：

- ユーザー名とパスワード
- スマートフォンや認証アプリ
- 顔認証や指紋認証

試験では、認証 = 本人確認、と押さえる。

---

### 認可

認可、または承認とは、認証されたユーザーに対して、どこまでアクセスを許可するかを決めること。

認証で本人確認ができても、認可で許可されていなければシステムやリソースにはアクセスできない。

試験では、認可 = 権限付与・アクセス許可、と押さえる。

---

### 認証と認可の違い

| 比較 | 意味 | 例 |
|---|---|---|
| 認証 | 誰なのかを確認する | ユーザー名とパスワードで本人確認 |
| 認可 | 何をしてよいかを決める | リソースの閲覧や変更を許可 |

覚え方：

- 認証：あなたは誰か
- 認可：あなたは何をしてよいか

---

## 3. Azure AD

### Azure ADとは

Azure ADは、Microsoftのクラウド認証サービス。

AzureやMicrosoft 365などのクラウドアプリへのサインインを管理する。

試験では次のように押さえる。

- クラウドの認証基盤
- Azure portalへのサインインに使われる
- Microsoft 365の認証にも使われる
- クラウドアプリへのシングルサインオンを提供する
- SalesforceやServiceNowなど、Microsoft以外のクラウドアプリとも連携できる

---

### シングルサインオン

シングルサインオン、SSOは、1回の認証で複数のクラウドアプリを利用できる仕組み。

試験では次のように押さえる。

- 1回サインインすれば複数アプリを利用できる
- ユーザーの利便性が上がる
- Microsoft 365やAzureで利用できる
- Microsoft以外のクラウドアプリとも連携できる
- アプリごとに何度もパスワード入力しなくてよい

---

### Azure ADによる認証の流れ

Azure ADによる認証では、セキュリティトークンが重要。

流れ：

- ユーザーがクラウドアプリにアクセスする
- Azure ADへリダイレクトされる
- ユーザーが認証情報を入力する
- Azure ADが認証を行う
- Azure ADがセキュリティトークンを発行する
- ユーザーはトークンを使ってクラウドアプリへアクセスする

一度認証されると、連携している他のクラウドアプリにはSSOでアクセスできる。

---

### Azure ADのユーザー管理

Azure ADには、ユーザーアカウントを登録できる。

主な登録項目：

- ユーザー名
- 名前
- 初期パスワード
- 役割
- 利用場所
- 役職
- 部署

Azure ADに登録されたユーザーは、ライセンスやアクセス許可があれば、連携するクラウドアプリにアクセスできる。

---

## 4. Azure ADとAD DS

### AD DSとは

Active Directory Domain Services、AD DSは、オンプレミス環境の認証サービス。

Windows ServerにAD DSの役割を追加すると、そのサーバーはドメインコントローラーになる。

試験では次のように押さえる。

- オンプレミスの認証サービス
- Windows Serverで提供される
- ドメインコントローラーがユーザーアカウントを管理する
- ドメイン参加したPCやサーバーを管理する
- オンプレミス環境でSSOを提供する

---

### Azure ADとAD DSの違い

Azure ADとAD DSは名前が似ているが、別物。

| 比較 | Azure AD | AD DS |
|---|---|---|
| 主な用途 | クラウドの認証 | オンプレミスの認証 |
| 対象 | Azure、Microsoft 365、クラウドアプリ | 社内PC、社内サーバー |
| SSO対象 | クラウドアプリ | オンプレミスサーバー |
| 主なプロトコル | OpenID Connect、OAuth2、SAML | Kerberos |
| 提供形態 | クラウドサービス | Windows Serverの役割 |

試験では、Azure AD = クラウド、AD DS = オンプレミス、と押さえる。

---

### Azure AD Connect

Azure AD Connect、現在のMicrosoft Entra Connectは、オンプレミスのAD DSユーザーをAzure ADへ同期するツール。

試験では次のように押さえる。

- AD DSとAzure ADのアカウントを同期する
- オンプレミス側で追加したユーザーをAzure ADへ同期できる
- ユーザーの二重管理を減らせる
- 同じIDでオンプレミスとクラウドアプリにアクセスしやすくなる
- 無償で提供されている

---

### Azure AD DS

Azure AD DS、現在のMicrosoft Entra Domain Servicesは、Azure上にAD DS相当の環境を提供するサービス。

Kerberos認証など、従来のAD DS機能が必要なアプリをAzureへ移行する場合に使う。

試験では次のように押さえる。

- Azure上でドメインサービスを利用できる
- Kerberos認証が必要なアプリに対応できる
- Microsoftがドメインコントローラーを管理する
- 利用者が自分でドメインコントローラーVMを構築しなくてよい
- オンプレミスから移行した古いアプリの互換性確保に使う

---

## 5. Azure ADの主な機能

### Azure ADのライセンス

Azure ADには、無償版と有償版がある。

| ライセンス | 特徴 |
|---|---|
| Azure AD Free | 基本的なユーザー管理やSSOが利用可能 |
| Microsoft 365アプリ | Microsoft 365利用者向けの機能を提供していたライセンス |
| Azure AD Premium P1 | 条件付きアクセスや動的グループなど高度な機能 |
| Azure AD Premium P2 | Identity Protectionなどさらに高度な機能 |

試験では、条件付きアクセスはP1以上、Identity ProtectionはP2と押さえる。

---

### 多要素認証

多要素認証、MFAは、複数の認証要素を組み合わせて本人確認する仕組み。

認証要素：

| 要素 | 例 |
|---|---|
| 知っているもの | パスワード |
| 持っているもの | スマートフォン、認証アプリ、ハードウェアトークン |
| ユーザー自身 | 顔、指紋などの生体情報 |

MFAを使うと、パスワードが漏えいしても、2つ目の要素がなければサインインできない。

---

### Azure AD MFA

Azure AD MFAは、Azure ADで利用できる多要素認証機能。

2要素目の例：

- SMSで確認コードを受け取る
- 電話で確認する
- Microsoft Authenticatorで承認する
- Microsoft Authenticatorのワンタイムパスコードを使う
- OATHハードウェアトークンを使う

試験では、MFA = パスワードだけに頼らず認証を強化する機能、と押さえる。

---

### 条件付きアクセス

条件付きアクセスは、認証後に、ユーザー・場所・デバイス・アプリなどの条件をもとにアクセスを制御する機能。

試験では次のように押さえる。

- 認証済みユーザーに対して追加でアクセス判断を行う
- 条件に応じて許可、拒否、MFA要求などを行う
- Azure AD Premium P1以上で利用可能
- ゼロトラストを実現する重要な機能

条件の例：

- 人
- 場所
- デバイス
- アプリ
- 許可 / 拒否

---

### 条件付きアクセスの例

| 条件 | 動作 |
|---|---|
| 管理者がAzure portalへアクセス | MFAを要求 |
| 社外から特定アプリへアクセス | ブロック |
| 管理対象デバイスからアクセス | 許可 |
| 社外からOffice 365へアクセス | MFAを要求 |

試験では、条件に応じてアクセスを細かく制御する機能と押さえる。

---

### Identity Protection

Identity Protectionは、ユーザーIDやサインイン動作のリスクを検出する機能。

Azure AD Premium P2で利用できる。

検出されるリスクの例：

- 通常起こりえない移動
- 普段とは異なるサインイン動作
- パスワードスプレー攻撃

試験では次のように押さえる。

- IDやサインインのリスクを検出する
- AIなどを使ってリスクを判定する
- リスクレベルは低・中・高で表示される
- リスクに応じてMFA要求やサインインブロックを自動化できる
- Premium P2の機能

---

## 6. Azureのアクセス管理

### RBAC

RBAC、ロールベースアクセス制御は、ロールを使ってAzureリソースへの管理権限を割り当てる仕組み。

試験では次のように押さえる。

- Azureリソースへの権限管理に使う
- ユーザーやグループにロールを割り当てる
- 必要な権限だけを与える
- スコープに応じて権限範囲が決まる

---

### ロール

ロールは、Azureリソースに対するアクセス許可のまとまり。

代表的な組み込みロール：

| ロール | 内容 |
|---|---|
| 所有者 | すべての操作が可能。ロール割り当ても可能 |
| 共同作成者 | リソースの作成・変更・削除が可能。ただしロール割り当ては不可 |
| 閲覧者 | リソースの設定値を閲覧できる。作成・変更・削除は不可 |

試験では、共同作成者はロール割り当てができない点に注意。

---

### カスタムロール

カスタムロールは、必要なアクセス許可だけを組み合わせて作成する独自ロール。

試験では次のように押さえる。

- 標準ロールで要件を満たせない場合に作成する
- 必要最小限の権限を定義できる
- JSON形式で定義される
- 最小権限の原則に役立つ

---

### ロール割り当ての3要素

ロールを割り当てるには、次の3つを組み合わせる。

| 要素 | 内容 |
|---|---|
| スコープ | 権限を適用する範囲 |
| ロール | 与える権限の種類 |
| セキュリティプリンシパル | 権限を与える対象 |

---

### スコープ

スコープは、ロールを割り当てる範囲。

Azureでは次の階層にロールを割り当てられる。

- 管理グループ
  - サブスクリプション
    - リソースグループ
      - リソース

上位階層に設定したロールは、下位階層へ継承される。

---

### セキュリティプリンシパル

セキュリティプリンシパルは、ロールを割り当てる対象。

例：

- ユーザー
- グループ
- サービスプリンシパル
- マネージドID

試験では、誰に権限を与えるかを表す対象と押さえる。

---

### IAM

IAM、Identity and Access Managementは、IDとアクセス権限を管理する仕組み。

Azure portalでは、各スコープの「アクセス制御（IAM）」からロールを割り当てる。

試験では、IAM = ロール割り当てを行う場所、と押さえる。

---

## 7. Azureのセキュリティ

### ゼロトラスト

ゼロトラストは、「決して信頼せず、常に検証する」という考え方。

従来は社内ネットワークの内側を安全と考えることが多かったが、クラウドやリモートワークではその前提が崩れる。

試験では次のように押さえる。

- 社内だから安全とは考えない
- 認証済みでも無条件に信頼しない
- アクセスのたびに状態を検証する
- 場所、デバイス、リスクなどを確認する
- 条件付きアクセスやIdentity Protectionが重要

---

### ゼロトラストと条件付きアクセス

条件付きアクセスを使うと、認証済みユーザーでもアクセス条件を検証できる。

例：

- 社内ネットワークからのアクセスは許可
- 社外からのアクセスはMFA要求
- 不明なデバイスからのアクセスは拒否
- 管理者操作には追加認証を要求

試験では、ゼロトラストを実現する代表的な機能として条件付きアクセスを押さえる。

---

### 多層防御

多層防御は、複数のレイヤーでセキュリティ対策を行う考え方。

1か所だけ守るのではなく、物理、ID、ネットワーク、アプリ、データなど複数の層で防御する。

主なレイヤー：

| 層 | 対策例 |
|---|---|
| 物理的なセキュリティ | 施錠、入退室管理 |
| IDとアクセス | 複雑なパスワード、アクセス権設定、MFA |
| 境界 | ファイアウォール、DDoS保護 |
| ネットワーク | 暗号化、不正パケット検知 |
| コンピューティング | OSパッチ、ウイルス対策、バックアップ |
| アプリケーション | アプリ制御、セキュリティパッチ |
| データ | 暗号化 |

試験では、多層防御 = 複数の層で守る、と押さえる。

---

## 8. Microsoft Defender for Cloud

### Microsoft Defender for Cloudとは

Microsoft Defender for Cloudは、Azure、オンプレミス、マルチクラウド環境のリソースを保護するためのセキュリティサービス。

試験では次のように押さえる。

- クラウド環境のセキュリティ態勢を管理する
- AzureだけでなくオンプレミスやAWSなども対象にできる
- セキュリティログを収集して評価する
- 機械学習や脅威インテリジェンスを活用する
- 推奨事項やアラートを提供する

---

### CSPM

CSPM、クラウドセキュリティ態勢管理は、クラウド環境の設定やリスクを評価する機能。

試験では次のように押さえる。

- セキュリティ設定を評価する
- 問題を検出する
- 推奨事項を提示する
- レポートを作成する
- セキュリティの高い環境作りを支援する

---

### CWP

CWP、クラウドワークロード保護は、実際に動いているサーバーやアプリなどのワークロードを保護する機能。

試験では次のように押さえる。

- 脆弱性チェック
- アンチウイルス
- ふるまい検知
- IDS
- IPS
- サーバー、ストレージ、SQL、コンテナー、App Serviceなどを保護

---

### Defender for Cloudのプラン

| プラン | 内容 |
|---|---|
| Foundational CSPM | 無料。既定で有効。推奨事項、セキュアスコアなどを提供 |
| Defender CSPM | 有償。脆弱性スキャン、攻撃パス分析など高度な機能 |
| Defenderプラン | ワークロードごとに有効化する有償保護機能 |

主なDefenderプラン：

- Defender for Servers
- Defender for Storage
- Defender for Azure SQL
- Defender for Containers
- Defender for App Service

---

### セキュアスコア

セキュアスコアは、Azure環境のセキュリティ状態を点数化する機能。

試験では次のように押さえる。

- セキュリティ状態を数値で確認できる
- 推奨事項を実施するとスコアが上がる
- 改善すべき項目を把握できる
- セキュリティ対策の優先順位付けに役立つ

---

### 推奨事項

推奨事項は、Azure環境のセキュリティを高めるために必要な対応を提示する機能。

例：

- Just-In-Time VMアクセスを有効にする
- 脆弱な設定を修正する
- セキュリティ機能を有効化する

推奨事項を順番に対応すると、セキュアスコアを改善できる。

---

### 規制コンプライアンスダッシュボード

規制コンプライアンスダッシュボードは、ISO 27001やNIST SP 800-53などの要件に対する準拠状況を確認する機能。

試験では、コンプライアンス要件への対応状況を一覧で確認できる機能と押さえる。

---

### セキュリティアラート

セキュリティアラートは、攻撃や脅威を検知して管理者へ通知する機能。

試験では次のように押さえる。

- システムログなどから攻撃を検知する
- Microsoftの脅威インテリジェンスを活用する
- 管理者に通知する
- セキュリティアラートマップで脅威発生源を視覚化できる

---

### Just-In-Time VMアクセス

Just-In-Time VMアクセスは、VMへの管理接続を必要なときだけ許可する機能。

RDPの3389番やSSHの22番は攻撃対象になりやすいため、普段は閉じておく。

試験では次のように押さえる。

- RDPやSSHのポートを普段は拒否する
- 必要なときだけ一時的に接続を許可する
- 許可時間を限定できる
- 不正アクセスのリスクを減らせる
- Defender for Cloudの機能として出やすい

---

## 9. 混同しやすい比較

### 認証と認可

| 比較 | 認証 | 認可 |
|---|---|---|
| 意味 | 誰かを確認する | 何を許可するか決める |
| タイミング | 先 | 後 |
| 例 | パスワード確認 | リソース閲覧を許可 |

---

### Azure AD / AD DS / Azure AD DS

| 比較 | Azure AD | AD DS | Azure AD DS |
|---|---|---|---|
| 用途 | クラウド認証 | オンプレミス認証 | Azure上のドメインサービス |
| 管理 | Microsoft管理 | 利用者が管理 | Microsoft管理 |
| 主な対象 | クラウドアプリ | 社内PC・サーバー | Azure上の従来型アプリ |
| キーワード | SSO、OpenID Connect | ドメインコントローラー、Kerberos | Kerberos対応、マネージドドメイン |

---

### MFA / 条件付きアクセス / Identity Protection

| 機能 | 役割 |
|---|---|
| MFA | 複数要素で認証を強化 |
| 条件付きアクセス | 条件に応じてアクセスを許可・拒否・MFA要求 |
| Identity Protection | IDやサインインのリスクを検出して対応 |

---

### 所有者 / 共同作成者 / 閲覧者

| ロール | できること | 注意点 |
|---|---|---|
| 所有者 | すべての操作、ロール割り当て | 最も強い権限 |
| 共同作成者 | リソース作成・変更・削除 | ロール割り当ては不可 |
| 閲覧者 | 閲覧のみ | 変更不可 |

---

### ゼロトラストと多層防御

| 比較 | ゼロトラスト | 多層防御 |
|---|---|---|
| 考え方 | 信頼せず常に検証 | 複数レイヤーで守る |
| 主な対象 | ID、場所、デバイス、リスク | 物理、ID、ネットワーク、アプリ、データ |
| 関連機能 | 条件付きアクセス、Identity Protection | Defender for Cloud、暗号化、パッチ、MFAなど |

---

## 10. 試験での聞かれ方

「ユーザーが誰かを確認する処理は？」  
→ 認証

「認証されたユーザーに権限を与える処理は？」  
→ 認可

「AzureやMicrosoft 365のクラウド認証に使うサービスは？」  
→ Azure AD / Microsoft Entra ID

「1回のサインインで複数アプリを使いたい」  
→ シングルサインオン

「オンプレミスのAD DSユーザーをAzure ADへ同期したい」  
→ Azure AD Connect / Microsoft Entra Connect

「Kerberos認証が必要なアプリをAzureへ移行したい」  
→ Azure AD DS / Microsoft Entra Domain Services

「パスワード以外の要素も使って認証したい」  
→ MFA

「社外アクセス時だけMFAを要求したい」  
→ 条件付きアクセス

「サインインリスクを検出し、自動対応したい」  
→ Identity Protection

「Azureリソースへの管理権限を割り当てたい」  
→ RBAC

「リソースの閲覧だけ許可したい」  
→ 閲覧者

「リソース作成は許可するが、ロール割り当てはさせたくない」  
→ 共同作成者

「すべての操作とロール割り当てを許可したい」  
→ 所有者

「認証済みでもアクセスのたびに検証したい」  
→ ゼロトラスト

「物理、ID、ネットワーク、データなど複数層で守りたい」  
→ 多層防御

「AzureやAWS、オンプレミスのセキュリティ状態を評価したい」  
→ Microsoft Defender for Cloud

「Azure環境のセキュリティ状態を点数で確認したい」  
→ セキュアスコア

「RDPやSSHを必要なときだけ開けたい」  
→ Just-In-Time VMアクセス

## 11. 最後に覚えるキーワード

- Azure AD：クラウド認証サービス
- Microsoft Entra ID：Azure ADの新名称
- 認証：本人確認
- 認可：アクセス許可
- SSO：1回の認証で複数アプリを利用
- セキュリティトークン：認証済みであることを示す情報
- AD DS：オンプレミスの認証サービス
- ドメインコントローラー：AD DSの認証サーバー
- Kerberos：AD DSで使われる主な認証プロトコル
- Azure AD Connect：AD DSとAzure ADを同期
- Azure AD DS：Azure上のマネージドなドメインサービス
- MFA：多要素認証
- 条件付きアクセス：条件に応じてアクセス制御
- Identity Protection：IDやサインインのリスク検出
- RBAC：ロールベースアクセス制御
- ロール：アクセス許可のまとまり
- 所有者：すべての操作とロール割り当てが可能
- 共同作成者：リソース操作は可能だがロール割り当て不可
- 閲覧者：閲覧のみ可能
- スコープ：ロールを適用する範囲
- 継承：上位階層の設定が下位に引き継がれること
- IAM：IDとアクセス権限の管理
- ゼロトラスト：信頼せず常に検証
- 多層防御：複数レイヤーで守る
- Microsoft Defender for Cloud：クラウドセキュリティ管理サービス
- CSPM：クラウドセキュリティ態勢管理
- CWP：クラウドワークロード保護
- セキュアスコア：セキュリティ状態の点数
- Just-In-Time VMアクセス：必要なときだけVM接続を許可
