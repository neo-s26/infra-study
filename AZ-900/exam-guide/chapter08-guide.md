# 第8章 Azureのガバナンス管理と監視｜試験型ガイド

## 1. この章で問われること

この章は、Azure環境をルールに沿って管理し、状態を監視するためのサービスを理解する章。

試験では、細かい操作手順よりも次の観点が問われやすい。

- ガバナンスとコンプライアンスの考え方
- Azure Policyの役割
- ポリシー定義とイニシアチブ定義の違い
- リソースロックの種類
- Azure Blueprintsで定義できるもの
- Service Trust Portalとトラストセンターの違い
- Azure Advisorの推奨事項
- Azure Service Healthの構成
- Azure Monitor、メトリック、ログ、アラートの役割
- Log AnalyticsとApplication Insightsの用途

---

## 2. 5分で見直す要点

### ガバナンス

ガバナンスとは、Azureリソースを適切に管理し、内部不正や人的ミスによるリスクを防ぐための仕組みづくり。

試験では次のように押さえる。

- Azureリソースの管理ルールを作る
- 組織のルールに沿ってリソースを運用する
- 誤操作や不正操作を防ぐ
- コストやセキュリティの統制に役立つ

---

### コンプライアンス

コンプライアンスとは、法令、業界基準、社内ルールなどに準拠すること。

Azureでは、次のような意味で出やすい。

- 公的機関や団体が定めた基準への準拠
- ISO 27001やPCI DSSなどへの準拠
- 会社が決めた運用ルールへの準拠
- リソース構成が基準を満たしているかの確認

---

## 3. Azure Policy

### Azure Policyとは

Azure Policyは、Azureリソースにルールを適用し、組織の基準に沿った運用を実現するサービス。

試験では次のように押さえる。

- リソース作成や構成にルールを適用する
- 許可されていない構成を拒否できる
- ルールへの準拠状況を確認できる
- コンプライアンス管理に使う
- 人的ミスやルール違反を防ぐ

---

### Azure Policyの使用例

Azure Policyでは、次のような制御ができる。

- 作成できる仮想マシンサイズを限定する
- 特定のリソース種類の作成を拒否する
- 許可されたリージョンだけを使用させる
- リソース作成時にタグ設定を強制する
- 組織のルールに合わないリソースを検出する

---

### ポリシー定義

ポリシー定義とは、Azure Policyにおける個々のルール。

例：

- 許可されたVMサイズのみ作成可能
- 特定リージョンのみ利用可能
- タグの付与を必須にする

試験では、ポリシー定義 = 1つ1つのルール、と押さえる。

---

### カスタムポリシー

カスタムポリシーは、組み込みのポリシー定義で要件を満たせない場合に作成する独自ポリシー。

試験では次のように押さえる。

- 組み込みポリシーで足りない場合に使う
- 組織独自のルールを定義できる
- Azure環境の統制に使う

---

### イニシアチブ定義

イニシアチブ定義とは、複数のポリシー定義をまとめたグループ。

複数のポリシーを1つずつ割り当てる代わりに、まとめて割り当てできる。

試験では次のように押さえる。

- 複数ポリシーをグループ化する
- まとめて割り当てできる
- コンプライアンス基準への準拠確認に使える
- ISO 27001やPCI DSSなどに対応する組み込みイニシアチブがある

---

### Azure Policyの適用範囲

Azure Policyは、次のスコープに割り当てできる。

- 管理グループ
  - サブスクリプション
    - リソースグループ

上位階層に割り当てたポリシーは、下位階層に継承される。

試験では、複数サブスクリプションに共通ルールを適用するなら管理グループに割り当てる、と押さえる。

---

### Azure Policyの動作

ポリシー適用後は、ルールに違反する新規作成や更新を制御できる。

ただし、ポリシー適用前から存在していた非準拠リソースは、自動削除されない。

試験では次のように押さえる。

- 新規作成や更新時に評価される
- 非準拠リソースを検出できる
- 既存リソースは自動削除されない
- 管理者が準拠状況を確認し、必要に応じて再構成する

---

### Azure Policyの修復

修復は、非準拠リソースをポリシーに準拠するように自動修正する機能。

例：

- タグがないリソースに自動でタグを付ける
- 必須設定がない場合に追加する

試験では、修復 = 非準拠リソースを準拠状態へ近づける機能、と押さえる。

---

## 4. リソースロック

### リソースロックとは

リソースロックは、重要なリソースの削除や変更を防ぐ機能。

人的ミスによる削除や変更を防ぐために使う。

試験では次のように押さえる。

- 重要リソースを保護する
- 誤削除や誤変更を防ぐ
- RBAC権限より優先して適用される
- 所有者でもロック中は制限を受ける
- 操作するには先にロックを削除する必要がある

---

### 削除ロック

削除ロックは、リソースの削除を禁止するロック。

試験では次のように押さえる。

- 削除操作を禁止する
- 作成や更新は可能
- 上位に設定すると下位リソースへ継承される
- 本番リソースの誤削除防止に使う

---

### 読み取り専用ロック

読み取り専用ロックは、リソースを読み取り専用にするロック。

試験では次のように押さえる。

- 閲覧のみ可能
- 作成、更新、削除ができない
- 削除ロックより強い
- リソース構成を変更させたくない場合に使う

---

### リソースロックの管理権限

リソースロックの作成や削除ができる主なロール：

- 所有者
- ユーザーアクセス管理者

同じ場所に複数のロックを設定することもできる。  
複数ロックがある場合、それぞれが単独で機能する。

---

## 5. Azure Blueprints

### Azure Blueprintsとは

Azure Blueprintsは、標準化されたAzure環境をサブスクリプションにまとめて展開するためのサービス。

Azure Policy、RBAC、ARMテンプレート、リソースグループなどをセットで定義できる。

試験では次のように押さえる。

- 標準化された環境を作成する
- サブスクリプション単位で割り当てる
- コンプライアンスに沿った環境を素早く構築できる
- 複数サブスクリプションに同じ構成を適用しやすい
- 追加料金は発生しない

---

### Azure Blueprintsで定義できるもの

Azure Blueprintsでは、次のものを定義できる。

- リソースグループ
- ARMテンプレート
- ロールの割り当て
- Azure Policyの割り当て

試験では、Blueprints = 環境構成のセット、と押さえる。

---

### Azure PolicyとAzure Blueprintsの違い

Azure Policyはルールを適用するサービス。  
Azure Blueprintsは、ルールや権限やリソース作成をまとめて標準環境として展開するサービス。

| 比較 | Azure Policy | Azure Blueprints |
|---|---|---|
| 目的 | リソース構成をルールで制御 | 標準環境をまとめて作成 |
| 主な対象 | ポリシー定義、イニシアチブ | Policy、RBAC、ARMテンプレート、リソースグループ |
| 用途 | 準拠状況の管理 | サブスクリプション環境の標準化 |

---

## 6. Service Trust Portalとトラストセンター

### Service Trust Portal

Service Trust Portalは、Microsoftクラウドサービスの監査レポートやホワイトペーパーなどを確認できるWebサイト。

試験では次のように押さえる。

- コンプライアンス関連資料を確認できる
- ISOやPCIなどの監査レポートを確認できる
- 資料ダウンロードにはサインインが必要
- 機密保持契約への同意が必要な場合がある
- マイライブラリで資料を保存できる

---

### トラストセンター

トラストセンターは、Microsoftクラウドサービスのセキュリティ、プライバシー、コンプライアンス情報を広く公開するWebサイト。

試験では次のように押さえる。

- 誰でもアクセスできる
- セキュリティ、プライバシー、コンプライアンス情報を確認できる
- 地域別や業界別のコンプライアンス認証を確認できる
- Microsoftクラウド全体の信頼性情報を確認できる

---

### Service Trust Portalとトラストセンターの違い

| 比較 | Service Trust Portal | トラストセンター |
|---|---|---|
| 主な内容 | 監査レポート、ホワイトペーパー | セキュリティ、プライバシー、コンプライアンス情報 |
| 資料ダウンロード | サインインが必要な場合あり | 基本的に公開情報 |
| 用途 | 詳細な証跡・監査資料の確認 | Microsoftクラウドの信頼性情報の確認 |

---

## 7. Azure Advisor

### Azure Advisorとは

Azure Advisorは、Azureのベストプラクティスに基づいて、リソース構成や運用に関する推奨事項を提示するサービス。

試験では次のように押さえる。

- Azureリソースの構成や利用状況を分析する
- ベストプラクティスに基づく推奨事項を提示する
- 推奨事項に従うかどうかは利用者が判断する
- 自動的に制限するサービスではない
- コスト、セキュリティ、信頼性、パフォーマンス、運用効率を改善する

---

### Azure Advisorの5カテゴリ

| カテゴリ | 内容 |
|---|---|
| コスト | 不要コスト削減、低使用VMの停止、予約購入など |
| セキュリティ | MFA、NSG、Defender for Cloud関連の推奨 |
| 信頼性 | バックアップ、レプリケーション、高可用性など |
| パフォーマンス | 応答速度や性能改善 |
| オペレーショナルエクセレンス | 運用管理や展開方法の改善 |

試験では、Azure Advisor = 推奨事項を出すサービス、と押さえる。

---

## 8. Azure Service Health

### Azure Service Healthとは

Azure Service Healthは、Azureサービスの障害、メンテナンス、リソース正常性などを確認するサービス。

試験では次のように押さえる。

- Azureサービスの正常性情報を確認できる
- 障害や計画メンテナンスを確認できる
- 自分のリソースへの影響を確認できる
- 正常性アラートで通知を受け取れる
- リソースを障害から保護するサービスではない

---

### Azure Service Healthの3要素

Azure Service Healthは、次の3つで構成される。

| 種類 | 内容 |
|---|---|
| Azureの状態 | Azure全体の大規模障害状況を公開するWebページ |
| サービス正常性 | Azure portalでサービス障害やメンテナンス情報を確認 |
| リソース正常性 | 個別リソースの正常性を確認 |

---

### Azureの状態

Azureの状態は、Azure全体のサービス停止状況を確認できる公開Webサイト。

試験では次のように押さえる。

- 誰でもアクセスできる
- Azure portalにログインできない場合にも確認できる
- リージョンごとのサービス状態を確認できる
- 大規模障害の確認に向く

---

### サービス正常性

サービス正常性は、Azure portalから自分の利用しているAzureサービスへの影響を確認するダッシュボード。

確認できるイベント：

- サービスに関する問題
- 計画メンテナンス
- 正常性の勧告
- セキュリティアドバイザリ

---

### リソース正常性

リソース正常性は、仮想マシン、App Service、ストレージアカウントなど、個別リソースの正常性を確認する機能。

試験では次のように押さえる。

- 特定リソースが正常か確認できる
- 問題がある場合は原因や対処情報を確認できる
- 自分が管理するリソース単位の正常性確認に使う

---

### 正常性アラート

正常性アラートは、Azureサービスの障害やメンテナンスなどを通知する機能。

通知条件の例：

- 対象サブスクリプション
- 対象リソース種類
- 対象リージョン
- イベント種類
- 通知先と通知方法

試験では、Azureサービス側の障害やメンテナンス通知を受け取りたい場合に使う、と押さえる。

---

## 9. Azure Monitor

### Azure Monitorとは

Azure Monitorは、Azureリソースやオンプレミス環境などを監視するためのサービス。

パフォーマンスやログを収集し、分析やアラート通知に利用できる。

試験では次のように押さえる。

- Azureの監視サービス
- Azureリソース、オンプレミス、他クラウドも監視対象にできる
- メトリックとログを収集する
- アラートを作成できる
- Application InsightsもAzure Monitorの一部

---

### メトリック

メトリックは、リソースのパフォーマンスに関する数値データ。

例：

- VMのCPU使用率
- App Serviceのサーバーエラー回数
- ストレージの使用量
- ネットワーク通信量

試験では次のように押さえる。

- 数値データ
- グラフ表示に向く
- Azure Monitorで既定収集される
- 無償で確認できる
- 既定の保持期間は90日

---

### ログ

ログは、システム内で発生したイベントのテキストデータ。

Azure Monitorで扱う主なログ：

- アクティビティログ
- リソースログ

試験では、ログ = 何が起きたかを記録するテキスト情報、と押さえる。

---

### アクティビティログ

アクティビティログは、Azureサブスクリプション内の管理操作ログ。

記録される内容：

- いつ
- 誰が
- 何を操作したか
- 成功したか失敗したか
- 変更履歴

試験では次のように押さえる。

- 管理操作のログ
- Azure Monitorが無償で収集
- 過去90日まで確認可能
- Azure ADのサインインログとは別物

---

### リソースログ

リソースログは、Azureリソース固有の情報を表すログ。

例：

- ストレージアカウントのBLOB読み取り
- BLOB削除
- 各リソース内部で発生したイベント

試験では次のように押さえる。

- リソースごとの詳細ログ
- 既定では収集されない
- 収集するには診断設定が必要
- 送信先としてLog Analyticsワークスペースやストレージアカウントを指定できる

---

### 診断設定

診断設定は、リソースログやメトリックの送信先を指定する設定。

送信先の例：

- Log Analyticsワークスペース
- ストレージアカウント

試験では、リソースログを収集・保存したい場合は診断設定を構成する、と押さえる。

---

### Log Analytics

Log Analyticsは、Azure Monitorログを分析するサービス。

ログやメトリックをLog Analyticsワークスペースに保存し、Kustoクエリで検索・分析する。

試験では次のように押さえる。

- ログ分析に使う
- Log Analyticsワークスペースにデータを保存する
- Kustoクエリで検索する
- Azure Monitorのログメニューからも実行できる
- 監査や詳細分析に向く

---

### Azure Monitorアラート

Azure Monitorアラートは、リソースで問題が発生したときに通知や自動アクションを実行する機能。

アラートルールでは、次を設定する。

- スコープ
- 条件
- アクショングループ

条件に使える主なシグナル：

| シグナル | 内容 |
|---|---|
| メトリック | CPU使用率などの数値データ |
| アクティビティログ | リソース操作ログ |
| ログ | Log Analyticsのデータ |
| Resource Health | リソース正常性情報 |

---

### アクショングループ

アクショングループは、アラート発生時に誰へどのように通知・対応するかを定義する。

通知方法の例：

- 電子メール
- SMS
- Azure Mobile Apps
- Azure Functionsなどによる自動処理

Azure Service Healthの正常性アラートでも使用できる。

---

### Application Insights

Application Insightsは、Webアプリケーションを監視するAzure Monitorの機能。

取得できる情報の例：

- アプリケーションの要求数
- 応答時間
- エラー回数や詳細
- 外部サービス呼び出し時間
- ユーザー分析情報
- サーバーのCPU、メモリ、ネットワーク使用率

試験では、Webアプリケーション監視に特化したサービスと押さえる。

---

## 10. 混同しやすい比較

### Azure Policyとリソースロック

| 比較 | Azure Policy | リソースロック |
|---|---|---|
| 目的 | ルールへの準拠を制御・評価 | 削除や変更を防止 |
| 例 | 特定リージョン以外を禁止 | 本番リソースを削除不可にする |
| 対象 | 構成ルール | 操作制限 |
| 既存非準拠リソース | 自動削除されない | ロック対象は操作不可 |

---

### 削除ロックと読み取り専用ロック

| 比較 | 削除ロック | 読み取り専用ロック |
|---|---|---|
| 削除 | 不可 | 不可 |
| 更新 | 可能 | 不可 |
| 新規作成 | 可能 | 不可 |
| 強さ | 弱め | 強め |

---

### ポリシー定義とイニシアチブ定義

| 比較 | ポリシー定義 | イニシアチブ定義 |
|---|---|---|
| 意味 | 1つのルール | 複数ルールの集合 |
| 例 | 許可リージョン制限 | ISO 27001対応の複数ルール |
| 用途 | 個別制御 | まとめて準拠管理 |

---

### Azure AdvisorとAzure Policy

| 比較 | Azure Advisor | Azure Policy |
|---|---|---|
| 目的 | 改善案を推奨 | ルールを適用・評価 |
| 強制力 | ない | ある |
| 例 | 低使用VMの停止を推奨 | 許可外リージョンの作成を拒否 |

---

### Azureの状態 / サービス正常性 / リソース正常性

| 比較 | Azureの状態 | サービス正常性 | リソース正常性 |
|---|---|---|---|
| 確認対象 | Azure全体 | 自分に関係するAzureサービス | 個別リソース |
| アクセス | 公開Webサイト | Azure portal | Azure portal |
| 用途 | 大規模障害確認 | 障害・メンテナンス確認 | 自分のVMなどの状態確認 |

---

### メトリックとログ

| 比較 | メトリック | ログ |
|---|---|---|
| 内容 | 数値データ | テキストデータ |
| 例 | CPU使用率 | 誰がVMを停止したか |
| 表示 | グラフ向き | 検索・分析向き |
| 用途 | パフォーマンス監視 | 操作履歴・イベント調査 |

---

### アクティビティログとリソースログ

| 比較 | アクティビティログ | リソースログ |
|---|---|---|
| 内容 | 管理操作ログ | リソース固有のログ |
| 例 | VM停止操作 | BLOB削除イベント |
| 収集 | 既定で収集 | 診断設定が必要 |
| 保持 | 既定90日 | 送信先設定による |

---

## 11. 試験での聞かれ方

「許可されたリージョンだけにリソースを作らせたい」  
→ Azure Policy

「特定サイズ以外のVM作成を拒否したい」  
→ Azure Policy

「複数のポリシーをまとめて割り当てたい」  
→ イニシアチブ定義

「既存の非準拠リソースは自動削除されるか」  
→ 自動削除されない

「タグを自動で付けて準拠状態にしたい」  
→ Azure Policyの修復

「本番リソースを誤って削除されたくない」  
→ 削除ロック

「リソースの作成・更新・削除を禁止し、閲覧だけにしたい」  
→ 読み取り専用ロック

「Policy、RBAC、ARMテンプレート、リソースグループをまとめて定義したい」  
→ Azure Blueprints

「Microsoftクラウドの監査レポートを確認したい」  
→ Service Trust Portal

「Microsoftクラウドのセキュリティやプライバシー情報を確認したい」  
→ トラストセンター

「低使用VMや孤立ディスクなどの改善提案を受けたい」  
→ Azure Advisor

「Azure全体の大規模障害を確認したい」  
→ Azureの状態

「自分が使うAzureサービスの障害やメンテナンスを確認したい」  
→ サービス正常性

「個別VMやApp Serviceの正常性を確認したい」  
→ リソース正常性

「CPU使用率などの数値データを確認したい」  
→ メトリック

「誰がいつVMを停止したか確認したい」  
→ アクティビティログ

「BLOB削除などリソース固有のログを集めたい」  
→ リソースログと診断設定

「ログをKustoクエリで分析したい」  
→ Log Analytics

「条件に応じてメールやSMSで通知したい」  
→ Azure Monitorアラート、アクショングループ

「Webアプリの応答時間やエラーを監視したい」  
→ Application Insights

## 12. 最後に覚えるキーワード

- ガバナンス：Azureリソースを統制・管理する仕組み
- コンプライアンス：法令、基準、社内ルールへの準拠
- Azure Policy：ルールを適用して準拠状況を管理
- ポリシー定義：個々のルール
- カスタムポリシー：独自に作成したポリシー
- イニシアチブ定義：複数ポリシーの集合
- 修復：非準拠リソースを準拠状態へ自動修正
- リソースロック：削除や変更を防ぐ機能
- 削除ロック：削除を禁止
- 読み取り専用ロック：閲覧のみ許可
- Azure Blueprints：標準化された環境をまとめて展開
- Service Trust Portal：監査レポートやホワイトペーパーを確認
- トラストセンター：セキュリティ、プライバシー、コンプライアンス情報
- Azure Advisor：ベストプラクティスに基づく推奨事項
- Azure Service Health：Azureサービスやリソースの正常性情報
- Azureの状態：Azure全体の障害状況を確認
- サービス正常性：自分に関係するサービス障害やメンテナンス
- リソース正常性：個別リソースの正常性
- Azure Monitor：Azureの監視サービス
- メトリック：パフォーマンスの数値データ
- ログ：イベントのテキストデータ
- アクティビティログ：サブスクリプション内の管理操作ログ
- リソースログ：リソース固有のログ
- 診断設定：ログやメトリックの送信先設定
- Log Analytics：ログの保存・分析サービス
- Kusto：Log Analyticsで使うクエリ言語
- Azure Monitorアラート：条件に応じた通知や自動アクション
- アクショングループ：通知先や実行アクションの定義
- Application Insights：Webアプリケーション監視
