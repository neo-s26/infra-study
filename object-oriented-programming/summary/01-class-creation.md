# 1.1 オブジェクトとは

## 1. この節の要点
- オブジェクト指向は、抽象的な説明だけだと掴みにくい。まずは**オブジェクトを作ってみる**のが理解への近道。
- **一番シンプルなオブジェクト = データの集まり**（関連する項目をひとまとめにしたもの）。
- オブジェクトを作るには、最初に**クラス（設計図）**を作る。クラスから複数のオブジェクトを作れる。

---

## 2. オブジェクトって何？
- ここでのオブジェクトは「商品1件分」みたいに、**意味的にまとまったデータを1つに束ねたもの**。
- 例：タブレット在庫表の1行（型番・品名・価格・発売日・在庫）を「商品オブジェクト」にする。

### なぜ “束ねる” のが大事？
- 1件分のデータには、文字列・数値・日付・真偽値など**型の違う値が混ざる**。
- 配列は基本的に「同じ型のデータを並べる」用途なので、こういう混在データの表現が苦しくなる。
- そこで、異なる型のデータをまとめられる**オブジェクト**が必要になる。

---

## 3. まずクラスを作る（クラスの役割）
- クラスは「プログラムの土台」ではなく、**オブジェクトをデザインする仕組み**。
- 先に「どんな項目を持つべきか（型番/品名/価格/発売日/在庫など）」を決めてクラスに並べる。
- クラスができれば、それを元に（コピーするイメージで）複数のオブジェクトを作れる。

---

## 4. クラスの作り方（Product例）
在庫表の項目を、クラスの変数として宣言する。

- number（型番）
- name（品名）
- price（価格）
- date（発売日：`LocalDate`）
- stock（在庫有無：`boolean`）

### ローカル変数とフィールド変数の違い
- **ローカル変数**：メソッドの中に書く。メソッド実行中だけ存在し、終わると消える。
- **フィールド変数（フィールド）**：クラスの中に書く。**オブジェクトの一部**として保持され、不要になるまで残る。

---

## 5. オブジェクトとして最低限必要な基本機能
クラスにフィールドを用意したら、次の3つを追加する（多くは自動生成できる）。

- **コンストラクタ**：オブジェクト生成時にフィールドへ初期値を入れる（`new` と一緒に使う）
- **ゲッター**：フィールドの値を取り出す
- **セッター**：フィールドの値を変更する

### mainメソッドはどうなる？
- クラスの本来の役割は「オブジェクトのデザイン」なので、設計段階では main は書かないことがある。
- ただし、**オブジェクトを作って使う側**では main（やそれに相当する起動処理）が必要。

---

# 1.2 クラスを作る

## 1. この節の要点
- Eclipseで「商品オブジェクト」を表す **Productクラス** を作り、最低限の形（フィールド＋コンストラクタ＋ゲッター/セッター）まで仕上げる手順を学ぶ。
- クラス作成の流れは **①フィールドだけ作る → ②コンストラクタ自動生成 → ③ゲッター/セッター自動生成**。
- フィールドは **private** を付けて外部から直接触れないようにし、操作はメソッド（ゲッター/セッター）経由にする。

---

## 2. クラスを作る手順（全体像）
1. **フィールド変数を持つだけのクラスを作る**
2. **コンストラクタを生成する**（自動生成）
3. **ゲッター・セッターを生成する**（自動生成）

補足：
- 章ごとのサンプルプロジェクト（ダウンロード済み）を使う。勝手に新規プロジェクトを作らない。  
  ※本用にパッケージやライブラリ、データファイルが用意されている場合があるため。

---

## 3. 手順① フィールドだけのクラスを作る（Product）
- `ch01` プロジェクトの `sample` パッケージに `Product` クラスを作成する。
- この時点では **mainメソッドは作らない**（クラスは「オブジェクトをデザインする」ためのものだから）。

### フィールド（在庫表の項目を写す）
在庫表の5項目を、そのままフィールドにする。

- 型番：`String number`
- 品名：`String name`
- 価格：`int price`
- 発売日：`LocalDate date`（日付型。必要なら import を追加）
- 在庫有無：`boolean stock`

重要：
- フィールドには **必ず `private` を付ける**  
  → 他のクラスから「見えない/触れない」状態にするため。  
  （クラス内のコンストラクタ、ゲッター、セッターからは使える）

---

## 4. 手順② コンストラクタを自動生成する
- フィールド宣言のあと（例：最終フィールドの次の行）にカーソルを置いてから生成する。
- Eclipseのメニューで **「ソース → フィールドを使用してコンストラクターを生成」** を選ぶ。
- ダイアログで **全フィールドにチェック** が入っていることを確認。
- 「`super()` 呼び出しを省略」系のチェックを入れてOK（説明は後で出てくる）。

注意：
- カーソルがクラスの外にあると「この操作は適用できません」系のエラーになる。

---

## 5. 手順③ ゲッター・セッターを自動生成する
- **「ソース → getter および setter の生成」** を選ぶ。
- **すべて選択**して全フィールド分を生成。
- 並び順は **「最初にgetter、次にsetter」** にする（読みやすくなる）。

---

## 6. 完成したクラスの読み方（this / static）
### this の意味
- コンストラクタなどで **引数名とフィールド名を同じにする**ことがある（例：`number`）。
- そのままだと区別できないので、**フィールド側に `this.` を付ける**。
  - 例：`this.number = number;`
- 引数が無いメソッド（getterなど）では、必ずしも `this` は必要ない。

### メソッドに static を付けない理由（この節の結論）
- `static` は「実行前にメモリへ用意しておく（静的に作る）」イメージ。mainが `static` なのは、最初に動く入口だから。
- 一方、オブジェクトは **実行中に必要な分だけ作る（動的に作る）** のが前提。
- だから、**オブジェクトをデザインするクラスのメソッドには static を付けない**。
- 同じ理由で、フィールド変数にも static が付いていない点に注意。
