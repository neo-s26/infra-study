# 2.1 インスタンスを作る

## 1. この節の要点
- **インスタンス（オブジェクトの実体）**は、`new` 演算子と **コンストラクタ** を使って作る。
- 作ったインスタンスは、ふつう **クラス型の変数**に代入して使う。
- `this` は「このクラスから作られたオブジェクト」を指し、**フィールド**と**引数**を区別するために使う。
- `new` はクラスから **フィールド変数とメソッドをコピー**して実体を作る。コンストラクタはコピーされず、**初期化のためだけに実行される**。
- クラスから作られた実体を **インスタンス** と呼ぶ。

---

## 2. インスタンスの作り方（new + コンストラクタ）
インスタンス生成はこの形が基本：

- `new Product( ... )` の `Product( ... )` 部分が **コンストラクタ呼び出し**  
- `...` は **コンストラクタの引数**（フィールドに入れる初期値）

例（在庫表の1行をもとに初期値を渡すイメージ）：
- 型番 `"A100"`
- 品名 `"XenPad"`
- 価格 `35760`
- 発売日 `LocalDate.of(2016, 9, 16)`
- 在庫 `true`

---

## 3. コンストラクタの役割と特徴
### 役割
- 受け取った **引数をフィールド変数へ代入**し、オブジェクトを初期化する。

### 特徴（重要）
- **new 演算子と一緒に使う**
- **クラス名と同じ名前**
- **メソッドではない**ので、**戻り値型を書かない**
- 主に **フィールド変数へ初期値を設定するため**に使う

補足：
- オブジェクトの「メンバ」は **フィールド変数とメソッド**の2つ。  
  コンストラクタは「初期化のための仕組み」で、メンバには含めない。

---

## 4. this とは（何を指す？なぜ要る？）
- `this` は **「このクラスのオブジェクト」**という意味。
- 典型例：コンストラクタ引数とフィールドを同じ名前にする（分かりやすいから）。
  - そのままだと区別できないので、**フィールド側に `this.` を付ける**
  - 例：`this.number = number;`

ポイント：
- 引数名を別にすれば `this` は不要になることもあるが、  
  **引数が多いほど同名の方が対応関係が分かりやすい**ので、同名＋`this` を採る。

---

## 5. 作ったインスタンスは「クラス型の変数」に代入して使う
生成しただけで終わらせず、通常はこう書く：

- `Product p1 = new Product(...);`

ここで大事なのは：
- 変数 `p1` の型は `Product`（= クラス名）
- **クラスは「設計図」だけでなく「型」でもある**  
  Javaではこの型を **クラス型** と呼ぶ（プリミティブ型と区別する）。

一般形：
- `型 変数 = new コンストラクタ`

---

## 6. 「インスタンス」と「オブジェクト」
- `new` とコンストラクタで作った実体は、どれも **クラスのインスタンス**。
- 「オブジェクト」は一般的な言い方、  
  「インスタンス」は **あるクラスから作られた実体（実例）**という言い方。

---

## 7. 複数のインスタンスを作る場所（main と実行用クラス）
- 何かを実行する処理は **mainメソッドに書く**のが基本。
- `Product` は「デザイン用」なので、実行処理は別クラスに書く方が自然。
- 例：同じ `sample` パッケージに `Exec` クラスを作り、そこで `main` を書いて `Product` を生成する。
  - 同じパッケージのクラス同士は互いに見える（visible）ので扱いやすい。

例題では `p1`〜`p5` の5つを作る：
- **同じ型（Product）**のインスタンスを何個でも作れる
- フィールドやメソッドの「形」は同じだが、**フィールドの値（属性値）は別**になり得る  
  → コンストラクタに渡す値が違うから

---

# 2.2 ゲッターとセッターの使い方

## 1. この節の要点
- インスタンスの「どのメンバ（フィールド/メソッド）」を使うか指定するには **ドット（`.`）= メンバ参照演算子** を使う。
- フィールドは `private` にして直接触らせず、外部からは **ゲッター/セッター経由**で値を取得・変更する。
- `private` にする目的は「意地悪」ではなく、**将来の仕様変更に強くする（変更を局所化する）**ため。
- 多くの場合 **ゲッターは必要**だが、**セッターは必須ではない**（値を変えない設計なら作らないのが普通）。
- セッターで値を変えるなら、不正値を入れないための **チェック** を追加する必要が出てくる。

---

## 2. メンバ参照演算子（ドット `.`）
同じクラスのインスタンス `p1`〜`p5` は、同じ名前のフィールド/メソッドを持つ。  
だから `getName()` だけだと「どのインスタンスの？」が決まらない。

- 正しい指定：`p1.getName()`（p1 の getName を呼ぶ）

書き方：
- `変数.メンバ`

補足：
- `p1.name` のようにフィールドへドットでアクセスする書き方自体は文法的に可能。
- ただしフィールドが `private` なので **他クラスからはアクセスできずコンパイルエラー**になる。

---

## 3. ゲッターの使い方
- **ゲッター**は「インスタンス内のフィールド値を返すメソッド」。
- 戻り値の型は、対応するフィールドの型と同じ。

例：
- `System.out.println(p1.getName());`（品名を表示）
- `System.out.println(p1.getPrice());`（価格を表示）

実装イメージ：
- `getName()` は `return name;`
- `getPrice()` は `return price;`

---

## 4. フィールドを private にする理由（本題）
もし外部のコードが `p1.number` のようにフィールドを直接使っていると、  
クラス設計を変えたときに **外部コードを全部探して書き換え**が必要になる。

例：製品番号を `number1` と `number2` に分割した場合  
- フィールド直接参照だと：利用側の修正が大量に発生
- ゲッター経由だと：`getNumber()` の中身を変えるだけで **利用側は修正不要**にできる  
  （例：`return number1 + number2;` のように返し方を調整できる）

結論：
- **外部にはメソッドだけ使わせる**（フィールドを隠す）と、変更に強くなる  
- そのためにフィールドは `private`、ゲッター/セッターは `public` にする

---

## 5. セッターの使い方
- **セッター**は「フィールド値を変更するメソッド」。

例：価格を変更して確認表示
- `p1.setPrice(40200);`
- `System.out.println("price: " + p1.getPrice());`

注意：
- `p1.price = 40200;` のような直接変更は **private なのでコンパイルエラー**。
- 値の変更はセッター経由で行う。

---

## 6. セッターは必須ではない
- 多くの場合ゲッターは必要だが、**セッターは必須ではない**。
- 「コンストラクタでセットした値は変更しない」方針のクラスでは、セッターを作らないのが普通。

さらに：
- セッターを積極的に使う場合、**不正な値を入れないチェック**（バリデーション）を追加する必要がある。

---

# 2.3 メソッドを追加する

## 1. この節の要点
- クラスには、必要に応じて **いくつでもメソッドを追加**できる。
- まずは「インスタンスの中身確認」に便利な **toString()** を作る（多くのIDEで自動生成できる）。
- `println(インスタンス)` のように書くと、内部で **toString() が呼ばれて**文字列表示される（自分で `p1.toString()` と書くより自然）。
- toString() を作らないと、デフォルト表示は「クラス名 + 位置っぽい値（ハッシュ）」になり、中身が分からない。
- 自動生成できないメソッドは手書きになるが、**ゲッター/セッターをお手本**にすると作りやすい。
- メソッドは「そのクラスが表すものに関係がある処理」を作る（例：Productなら価格×個数の総額）。

---

## 2. toString() メソッドを作る（自動生成）
### 目的
- フィールドの値をまとめて見やすい文字列にして返す  
  → 動作確認がラクになる（全部ゲッターで1個ずつ出すのは面倒）

### Eclipseでの作り方（流れ）
1. Productクラス末尾にカーソル
2. メニュー：`ソース` → `toString() の生成`
3. ダイアログで **全フィールドがチェック**されているのを確認してOK

生成される内容のイメージ：
- `Product[number=..., name=..., price=..., date=..., stock=...]` みたいな形の文字列を返す

補足：
- `@Override` は **アノテーション**。ここでは消しても動くが、重要な要素なので基本は残しておく（後で詳しく出る）。

---

## 3. toString() を使う正しい考え方
- `System.out.println(p1.toString())` でも表示できるが、
- 正解は `System.out.println(p1)`  
  → println側が、表示用に **自動で p1.toString() を呼ぶ**仕組みになっている

### toString() の意義
- **printlnなどの出力系メソッドが利用するため**に用意するメソッド、という理解が大事。

---

## 4. 既定（デフォルト）の toString() だと役に立たない
- toString() を作らない状態で `println(p1)` すると、
  - `sample.Product@15db9742` のような表示になる
- `@...` の部分は、インスタンスのメモリ上の位置に基づく値（っぽいもの）で、
  Javaでは基本的に役に立たない（中身は分からない）

結論：
- **インスタンスをちゃんと表示したければ、自分で toString() を作る**必要がある。

---

## 5. メソッドを増やすには（手書きの基本形）
- すべてが自動生成できるわけではない。必要に応じて手で書く。
- ゲッターは「値を返す（return）」、セッターは「値を受け取って代入する」。
  これを組み合わせると普通の処理メソッドが書ける。

一般形：
- `public 戻り値型 someMethod(引数...) { ... return ...; }`

---

## 6. 例：Productに totalPrice(quantity) を追加する
### 何をする？
- 個数 `quantity` を受け取り、`price * quantity` を計算して **総額を返す**

ポイント：
- メソッドは「Productという概念に妥当な機能」にする  
  （意味のないメソッドを増やすのはNG）

実装イメージ：
- 引数：`int quantity`
- 戻り値：`int`
- 計算：`int total = price * quantity; return total;`

### 使い方（例）
- `p1.totalPrice(quantity)` を呼び出して総額を表示する
- ゲッターで品名を取って表示する、などと組み合わせて動作確認できる

---

## 7. この節のまとめ
- toString() を用意すると、インスタンスの中身確認が一気に楽になる
- 出力は `println(p1)` が自然（内部で toStringが使われる）
- クラスのメソッドは、必要に応じて追加できるが「クラスの目的に沿う処理」にする
- 手書きメソッドは、ゲッター/セッターを参考に「受け取る→処理→返す」で作れる
